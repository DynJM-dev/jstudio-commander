## Tests-to-user-observable-contract (per C6 carryover from v5)

New tests MUST pin user-observable DOM strings, not internal function return values. Bad: `expect(getActionInfo(msgs)).toEqual({label: 'Reading X...', icon: ...})`. Good: renders the full prop stack (ChatPage → ContextBar) and asserts on the status-bar `<span>` textContent.

Minimum test set for ship:

**Item 1 tests (integration framing, §20.LL-L10 compliant):**

1. Given a ChatMessage tail `[user, assistant(tool_use(Read,"X.md")), assistant(text("I see..."))]` and `isWorking=true` (via `isWorkingOverride`), ContextBar status-bar DOM reads `"Reading X.md..."`. (E0.2 regression pin — the reverse-scan finds the Read even with a trailing text block.)
2. Given parallel tool_uses in one assistant message `[tool_use(Read,A), tool_use(Read,B)]`, DOM reads `"Reading B..."` (most-recently-dispatched wins, matching Codeman; reverse-scan stops at first tool_use which is the last-written).
3. Given a thinking block tailing a still-unmatched tool_use `[tool_use(Edit,"foo.ts"), thinking("...")]`, DOM reads `"Editing foo.ts..."`. (E0.3 regression pin.)
4. Given a stale-`lastActivityAt` (>30s) + unmatched `tool_use` window, DOM continues to read the tool's rich label for the full window (regression guard against the E0.3 regression-to-`"Working..."` via `shouldSuppressComposingLabel` path).
5. **Bash Case 1 non-regression**: given `[user, assistant(tool_use(Bash,"sleep 10 && echo done"))]` and `isWorking=true`, DOM reads `"Running command..."`. Asserted before AND after the `getActionInfo` change.

**Item 3 tests:**

1. Given `session.status='idle'` on the client AND a prompt is present server-side, `<PermissionPrompt>` mounts within 3s of the initial page load (no refresh required). User-observable: the modal DOM node appears.
2. Given a session that force-idled mid-approval-window, the modal eventually appears without a browser reload action.
3. Regression: idle sessions without prompts do NOT show a mounted `<PermissionPrompt>` (false-positive guard).
4. Cadence test: when `sessionStatus` is active, polling fires at ≤ 2s intervals; when idle, polling fires at 5–10s intervals. Mock timers acceptable; pin the observable interval, not the timer internals.
