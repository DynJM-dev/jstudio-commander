## Why instrumentation, not another fix

Three prior fix rotations on 15.3 have passed unit tests, shipped green-by-tests, and failed live browser smoke. Pattern is hypothesis-first reasoning with source-reading as primary evidence. Per `feedback_understand_before_patching.md` and `feedback_debug_with_real_data`, the discipline now is runtime evidence first. We stop patching until we understand the system at the level of actual values flowing through actual function calls at actual render moments.

Critical live observation from Jose 2026-04-20 ~02:51 AM: in two concurrent active CODER sessions, the activity monitor IS rendering rich labels correctly right now — "Running command... 3s" for Bash, "Thinking deeply... 27s", "Mulling 28s", "Choose an option 8s". Yet Tier A smoke 15 minutes earlier on the same Commander build showed Read STATE.md rendering as "Idle". This is intermittency-class, not absolute absence. The instrumentation must capture BOTH a working case and a failing case in the same session — without a working control, the log diff has no reference point.

## Capture protocol — five cases, one continuous session

CODER runs ALL five cases sequentially in a single session with DevTools console open throughout. Do NOT restart Commander between cases. Do NOT clear the console between cases — timestamps across cases matter. If the console overflows, export early and continue.

Case 1 — BASH-10. Jose prompts CODER's session with `sleep 10 && echo done`. CODER dispatches the Bash tool_use. Capture the full 10-second window: every `[15.3-instr]` log emitted during the tool's execution, from dispatch to result.

Case 2 — READ-STATE. Immediately after BASH-10 completes, Jose prompts CODER with "Read ~/Desktop/Projects/jstudio-commander/STATE.md". CODER dispatches a Read tool_use. Capture the window from dispatch to result.

Case 3 — EDIT-DIAG. Immediately after READ-STATE completes, Jose prompts CODER with "Edit docs/15.3-diagnostic.md to append a single newline". CODER dispatches an Edit tool_use. Capture the window from dispatch to result.

Case 4 — SPAWN-AGENT. Immediately after EDIT-DIAG completes, Jose prompts CODER with "spawn an Explore agent to grep for TodoWrite in the client tree". CODER dispatches a Task tool_use spawning an Explore subagent. Capture from dispatch to subagent completion.

Case 5 — BASH-2. Immediately after SPAWN-AGENT completes, Jose prompts CODER with `sleep 5`. CODER dispatches Bash. Capture. This is the control for whether Bash still works after the session has accumulated other cases, or whether Bash degrades across the session.

If any single case fails to produce meaningful logs (browser crash, Commander hang), note that explicitly in §12.1 and continue with remaining cases. A failed case IS data.

Equal importance: if all five cases render rich labels correctly, that is ALSO data — it means the bug did not reproduce during this capture, and the failure class is narrower than we thought. CODER reports honestly without contorting the framing to match a pre-supposed narrative. "All 5 passed" is an acceptable §12 conclusion that triggers a different PM conversation.
