## Fix 1 — widen `isSessionWorking` to consume typed `Working` with a freshness predicate

**Location:** `client/src/pages/ChatPage.tsx:329-332`.

**Current shape:**
```
isSessionWorking = userJustSent
  || (session?.status === 'working' && !heartbeatStale)
  || unmatchedToolUse
```

**New shape (contract):**
```
isSessionWorking = userJustSent
  || (session?.status === 'working' && !heartbeatStale)
  || unmatchedToolUse
  || (sessionState?.kind === 'Working' && sessionStateIsFresh)
```

Where `sessionStateIsFresh` means: the current `sessionState` value was observed by the client AFTER the most recent user message was sent. This closes Case 2 — when `session.status` lags but `sessionState.kind === 'Working:*'` is emitted fresh for the current turn, the bar flips working within the poll cadence instead of waiting 15-20 seconds for `session.status` to catch up.

**Staleness mitigation — load-bearing, NOT optional.** Without the freshness gate, a stale-Working carryover from a prior turn (e.g., server hasn't reset typed state between turns) would pin the bar as Working during genuine idle periods. The freshness gate makes the OR-branch safe.

**Implementation shape (pick what fits cleanest):**

Option A — additive hook `useSessionStateUpdatedAt(sessionId)` that returns the wall-clock timestamp of the last time `useSessionState` delivered a new value. Uses `useRef` + `useEffect` on the current hook's ref. Compare against `lastUserMessageTs` (derived from `messages`, find the most recent `role === 'user'` message and read its timestamp).

Option B — change `useSessionState`'s return shape to `{ state, updatedAt }`. One call site (ChatPage.tsx:56). Slightly larger surface; destructure at call site.

CODER picks. Either is fine. Option A is additive and slightly safer; Option B is one-hook and slightly cleaner.

**Edge case to test:** if the user sends a message but `sessionState` hasn't changed yet (because the server hasn't emitted a new typed-state event in response), `sessionStateUpdatedAt < lastUserMessageTs` and the freshness gate returns false. That's fine — the `userJustSent` OR-branch carries the bar for the first 2-3s while the server catches up. Freshness gate fires once typed state updates for this turn.

## Fix 2 — `resolveActionLabel`'s typed:Idle branch falls through when `isWorking && jsonlLabel`

**Location:** `client/src/utils/contextBarAction.ts:178-181` (typed:Idle case in the switch).

**Current shape:**
```
case 'Idle':
  return null;
```

**New shape (contract):**
```
case 'Idle':
  // Stale-typed-Idle can clobber a live jsonlLabel during a fresh Working
  // turn. When the bar is actively working (isWorking=true from Fix 1's OR-chain)
  // and we have a rich jsonlLabel from getActionInfo, fall through to the
  // jsonlLabel rather than returning null. Preserves true-idle render when
  // no Working signal is present. Extends the §6.1 inversion.
  if (isWorking && jsonlLabel) return jsonlLabel;
  return null;
```

This closes Case 3 — when stale `Idle:AwaitingFirstPrompt` is the typed state but `getActionInfo` has already produced `"Running command..."` from the JSONL, the rich label reaches the DOM. Preserves true-idle (no jsonlLabel, not working → still returns null → `getStatusInfo` Idle branch fires correctly).

**Coupling with Fix 1:** Fix 1 widens WHEN `isWorking === true`; Fix 2 widens WHICH label renders while working. Neither alone closes both Case 2 and Case 3.
