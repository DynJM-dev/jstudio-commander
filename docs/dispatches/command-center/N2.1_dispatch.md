## §1 — What's broken

Per PHASE_N2_REPORT §7 Debt 15. `~/.commander/config.json`'s bearer token is rewritten across sidecar launches without explicit trigger. Evidence: CODER smoke-readiness 2026-04-23 ~10:15 UTC used bearer `5f77f209-…`; Jose MCP test ~16:40 UTC used bearer `69b70ef5-…` with `config.json.updatedAt = 2026-04-23T15:59:03.816Z`. Sidecar rewrote the config somewhere between those timestamps.

**Contract being violated:** D-N1-07 §8.2 — "Single local bearer token at `~/.commander/config.json`. v1: no expiry." Implied mint-once-on-first-launch + persist.

**Downstream impact:** N3 long-running agent runs with external MCP sessions holding a hardcoded bearer will silently 401 mid-run after rotation. Any published setup docs that include "paste your bearer here" have a half-life of hours. Both behaviors are unacceptable before N3 introduces the long-lived run surface.

**G10 applies in spirit** but not as a full instrumentation rotation: PM has already provided the symptom evidence + timing window. CODER's first action is source-grep to enumerate write paths, then trace which one fires unintentionally. That grep + trace IS the instrumentation. No separate rotation required unless the first fix ships unit-green + symptom unchanged.

## §2 — Scope

Three tasks. No Rust changes. No new deps expected.

**T1 — Enumerate every `config.json` write path in the sidecar.** Source-grep for:
- Writes to `config.json` (any file write whose path resolves to it).
- Every generator of bearer-like values (`crypto.randomUUID()` calls in config-adjacent code paths).
- Every call site that mutates the in-memory config object before a write.

Document the full list in PHASE_N2.1_REPORT §5 (issues + resolutions). Expected paths based on PM's §3 root-cause candidates: first-launch initializer, boot-time validator, client-disconnect re-init, port-scan retry. Actual list may differ — don't assume.

**T2 — Identify the unintentional write and fix it.** Two acceptable fix shapes; CODER picks based on what T1 reveals:

**Option A — Preserve on every launch.** If `config.json` exists AND parses cleanly AND has a valid UUID bearer field, the sidecar MUST preserve it. Only mint a fresh bearer when the file doesn't exist OR fails to parse OR is missing the bearer field. Any other path that currently rewrites bearer is a bug — fix it to preserve.

**Option B — Intentional rotation is documented.** If T1 reveals the rewrite is intentional (e.g., boot-time security hygiene, some specific trigger we want to keep), document the trigger in D-N1-07 §8.2 amendment + add a `lastRotatedAt: ISO-8601 string` field to `config.json` + emit a WS `system:warning` event on rotation carrying the new bearer so external sessions can refresh. Then ensure rotation does NOT happen on the uneventful relaunch path that caused this bug.

**PM lean, CTO concurs:** Option A is almost certainly correct. "Mint once, never rotate in v1" matches the D-N1-07 §8.2 contract as written. Option B is the escape hatch if T1 reveals rotation is wired to something load-bearing that we don't want to rip out. Default to A; justify B in PHASE_REPORT §4 if you go that way.

**T3 — Regression test.** `apps/sidecar/tests/integration/bearer-persistence.test.ts` (bun:test):

1. Spin up sidecar fresh (no `config.json` present). Read generated bearer. Shut down.
2. Spin up sidecar again against the same `~/.commander/`. Read bearer. Assert equal to step 1's bearer.
3. Spin up sidecar a third time. Assert still equal.
4. Delete `config.json`. Spin up sidecar. Assert new bearer (different UUID). Persistence only kicks in when config exists.
5. Corrupt `config.json` (write `"not valid json"` to it). Spin up sidecar. Assert new bearer AND corruption handled without sidecar crash.

Test uses a temp `~/.commander/` path (isolate from Jose's real state). Clean up after.
