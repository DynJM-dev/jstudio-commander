## 3. Pipelines that feed Commander's UI state

There are three independent pipelines. The bug class lives in their composition, not in any single stage.

### 3.1 Pane-regex pipeline — `status-poller.service.ts` (5s interval)

```
tmux capture-pane
  → classifyStatusFromPane()      (pane text regex — `❯` idle, verb footer, numbered-choice, etc.)
  → applyActivityHints()          (15.1-D allowlist; upgrades idle→working)
  → [override] hasPendingToolUseInTranscript(transcriptPath)  when hinted ∈ {idle, waiting}
  → computeSessionState()         (typed SessionState per Phase 1)
  → eventBus.emitSessionStatus(id, 'working'|'idle'|…, { state: SessionState, … })
```

**Key constants (status-poller.service.ts:11–90):**
- `POLL_INTERVAL = 5_000` — hard ceiling on how quickly this pipeline can react to a new tool_use.
- `HOOK_YIELD_MS = 60_000` — if any hook fired within 60s and the pending-tool override did NOT fire, skip all reclassification.
- `FORCE_IDLE_COOLDOWN_MS = 60_000` — if row was stale-force-idled recently, skip EVERY reclassification including the pending-tool override. **This is a trap** (§5 D3).
- `STALE_ACTIVITY_MS = 90_000` — stuck-working force-idle trigger.
- `IDLE_GRACE_MS = 8_000` — working→idle only after 8s grace.

### 3.2 Hook pipeline — `hook-event.routes.ts` (push, real-time)

SessionStart / Stop / PostToolUse / SessionEnd → `processHook`:
- SessionStart → `status='working'`, bump heartbeat (:385)
- Stop → gated idle flip on `hasPendingToolUseInTranscript` (:344); emits `eventBus.emitSessionStatus(…, 'idle', {evidence:'stop-hook'})` when gate false.
- Every hook → `bumpLastActivity` + `bumpLastHookAt` (drives `last_hook_at` used by §3.1's yield gate).

### 3.3 Session-tick pipeline — `session-tick.service.ts` (push, statusline-driven)

The installed statusline hook posts `/api/session-tick` every time Claude Code repaints its statusline (seconds-ish cadence while Claude is drawing). Commander upserts `session_ticks`, emits `session:tick`. Client `useSessionTick` feeds ContextBar's `resolveContextPercent` (tick wins, S.1), LiveActivityRow extras, HeartbeatDot freshness.

**The gotcha here** (self-evident but load-bearing for the diagnosis): Claude Code does NOT repaint the statusline during tool execution. During `sleep 10` the statusline stream falls silent for the full 10 seconds. Any UI that treats `session_ticks.updated_at` as proof-of-life will read stale during every tool call.

## 4. Code-level drop-points (class root cause)

### 4.1 The controlling drop: `pendingToolName` is never plumbed.

- `hasPendingToolUseInTranscript(path): boolean` (jsonl-parser.service.ts:669) returns **only a boolean**. It does not return the tool name.
- `status-poller.service.ts:402–404` passes `pendingToolUse: boolean` to `computeSessionState` — and nothing else.
- `session-state.service.ts:167–173` constructs `Working:ToolExec` with `toolName: undefined` whenever the override fires.
- `client/src/utils/contextBarAction.ts:124–126` maps `Working:ToolExec` without `toolName` to the **generic** string `'Running tool…'`.

**Consequence.** Acceptance criterion #2 ("Running Bash not Working") is structurally impossible to satisfy via the typed-state pipeline as wired. Every pending-tool override produces the generic label; the richer "Reading STATE.md…" / "Editing x.ts" / "Running Bash" labels that `client/src/components/chat/ContextBar.tsx:48–68` can already derive from `ChatMessage[]` are **blocked** because `resolveActionLabel` returns `'Running tool…'` and does NOT fall through to `jsonlLabel` for the ToolExec case (see `contextBarAction.ts:123–135`: only `Working:Generic` falls through; `Working:ToolExec` returns unconditionally).

This is the textbook **policy inversion** pattern from memory `feedback_policy_inversion_default_render.md`. Phase 1 promoted server-typed state to authoritative; the server's typed-state producer is **impoverished** relative to the legacy client-side derivation. Result: rich labels masked by a generic one.

## 5. Class root cause (single explanation)

**Phase 1 inverted the policy without moving the information.**

Before Phase 1: ContextBar's `getActionInfo(messages)` walked the last assistant block on the client and produced a rich, tool-name-aware label ("Reading STATE.md…", "Running command…") from the ChatMessage stream that `useChat` already maintained. Legacy, slightly inconsistent across surfaces, but **information-rich**.

After Phase 1: the authoritative source became server-computed `SessionState`. The server's producer (`computeSessionState` + `hasPendingToolUseInTranscript`) operates on **boolean presence** of a pending tool_use and does NOT carry the tool name, file path, or any of the richer signal the client had. The client's rich `getActionInfo` still exists but is now **gated off** by `resolveActionLabel` returning a generic string for `Working:ToolExec`.

Compounding this, every timing constant in the pane pipeline is tuned to **suppress churn** (8s grace, 60s yield gate, 60s force-idle cooldown, 90s stale-force-idle). Those gates were added one at a time to fight specific oscillation bugs; stacked, they produce multi-second-to-multi-minute windows where the server's view of session state cannot change even when JSONL tells a different story.

The **single coherent class root cause** is: _Commander treats "compute session state" as a server-owned, poll-driven, impoverished pipeline gated against churn — but the user-experience acceptance criteria demand a client-owned, event-driven, information-rich pipeline gated only against visible wrongness._ Every symptom Jose reports reduces to a direct consequence of this inversion: the server is too slow, too coarse, and too conservative to be authoritative, but Phase 1 promoted it anyway. Phase 1.1 widened the override to fire from `waiting`, which is correct but still leaves four other drop-points (D1–D5) and the controlling name-plumb drop (§4.1).

## 6. Proposed fix shape (contract-level — no code, no file:line patches)

**Principle.** Invert the invert. The **client's rich derivation** is the authoritative label source for `Working` (and nested subtypes like ToolExec / Composing / Thinking). The **server's typed state** is authoritative for coarse lifecycle transitions (Stopped, Error, WaitingForInput, Compacting, Idle-vs-Working) where the client has no better signal than the server.

This is a reversal of Phase 1's Working-path priority **without** giving up the structural wins of the typed state enum (type safety, Waiting subtype differentiation, Compacting signal, future sub-agent subtype).

### 6.1 Label-source priority, new

For a session whose coarse status is `working`:

```
1. Client jsonlLabel from getActionInfo(messages)     ← authoritative for Working
2. Typed-state subtype hintLabel (Thinking verb)
3. terminalHint (legacy, Compacting literal)
4. Generic "Working..." fallback
```

For Waiting, Stopped, Error, Compacting, Idle: typed state wins (unchanged).

Implementation contract: `resolveActionLabel` inverts the `Working` branch so `jsonlLabel` is consulted before `Working:ToolExec`'s `'Running tool…'` generic. The `Working:ToolExec` case remains in the enum for correctness of reporting and DevTools but is no longer the UI label source.

### 6.2 `isWorking` derivation, new

Today `isWorking` gates LiveActivityRow + ContextBar animation. Derive it from **two OR conditions**:
  - Server says `session:status` coarse status is `'working'`.
  - Client observes an **unmatched tool_use** in the latest `ChatMessage[]` (scan the tail for `tool_use` not followed by a `user.tool_result` with matching id).

The client-side detection is symmetric to the server's `hasPendingToolUseInTranscript` but runs on the ChatMessage stream the client already has, updating immediately on `useChat`'s poll — no waiting for the 5s poller + WS round trip. The server's `isWorking` still wins for post-turn-end recovery (client's tool_use scan won't detect `Composing` without explicit plumbing, which is fine — Composing isn't a tool).

### 6.3 Heartbeat staleness for LiveActivityRow, new

LiveActivityRow's `heartbeatStale` predicate should exclude the tool-execution window. Today it treats "no statusline tick in 30s" as "probably dead". During a single long tool call (`pnpm test`, a long grep, network request) the statusline goes silent **intentionally**. The predicate should be:

```
heartbeatStale := (no statusline tick in 30s) AND (no unmatched tool_use in ChatMessage tail)
```

So the bubble indicator stays live for the full duration of any tool call, regardless of statusline silence.

### 6.4 Poller timing re-tune (optional, secondary)

Not strictly required by the 10 criteria once §6.1–6.3 land — the client-side path makes server latency invisible on the label. But for defense-in-depth on criterion #5 (no backward flashes):
  - Drop `POLL_INTERVAL` to 1–2s **OR** tighten the override to run on every hook event, not just poller ticks.
  - Narrow `FORCE_IDLE_COOLDOWN_MS` to not suppress the pending-tool override (the override is the one signal we trust more than the cooldown).

These are optional and should land in a separate rotation after §6.1–6.3 smoke green.

### 6.5 Regression surfaces to watch

- **Phase N.0 P1 Stop-hook gate** — still correct; the gate on `hasPendingToolUseInTranscript` keeps the 'idle' flip suppressed during tool exec. Do not touch.
- **Phase J.1 IDLE_VERBS allowlist** — still correct; the `working` verb classification is not on the path we're changing.
- **Phase L B1 past-tense gate** — still correct; only affects pane-derived status, not the new client-side label path.
- **Phase M statusline forwarder / Phase S.1 tick-first ContextBar** — the `resolveContextPercent` tick-first invariant is unaffected. Context %, not label.

- **Sub-agent monitoring** — out of scope for the minimum fix; a follow-up rotation can add `Idle:MonitoringSubagents` → "Monitoring N subagents" label if desired (criterion #6 partial).

### 11.1 Item 1 — Read/Edit/Write label intermittency (class: "tail-block-only label derivation")

**E0 citation:** E0.2 proved the rich-label render path works (`"Reading PHASE_V_BRIEF.md · 74s · Long task"`). E0.3 + E0.5 proved it regresses mid-turn to generic `"Working..."`. Intermittency, not absence.

**Class root cause (source-level):**

`getActionInfo` at `client/src/components/chat/ContextBar.tsx:34–73` scans to the last assistant message, then derives its label from **exactly one block** — `blocks[blocks.length - 1]` at `:42–43`. The Read/Edit/Write/Bash tool-name branches at `:49–69` apply only when that terminal block's `type === 'tool_use'`.

When the last block of the last assistant message is **not** a tool_use — because (a) Claude emitted text or thinking AFTER dispatching a tool_use within the same assistant message, (b) a later assistant message arrived with a text/thinking tail while an earlier tool is still pending, or (c) the parser's consecutive-assistant grouping landed a text block at the tail — the function returns either `'Composing response...'` (text), `'Cogitating...'` (thinking), or `null` (no content). The tool's still-running identity is present in an **earlier** block, but `getActionInfo` never walks backward past the last block.

File:line evidence of the single-block scan limit:
- `ContextBar.tsx:42` — `const lastBlock = blocks[blocks.length - 1];`
- `ContextBar.tsx:44` — `if (!lastBlock) return null;` (no iteration, no unmatched-tool_use lookup)
- `ContextBar.tsx:49` — `if (lastBlock.type === 'tool_use')` — tool-name branches only fire on tail position.

**Parallel-tool JSONL evidence from coder's own session (`94f87c69-c670-4455-b886-df70a3910521.jsonl`):**

```
line 17  2026-04-20T05:42:21.857Z  tool_use(Read)
line 18  2026-04-20T05:42:21.887Z  tool_use(Read)   — 30ms later, separate assistant record
```

Two Reads dispatched 30ms apart. Each is a distinct assistant JSONL record, so `useChat`'s `ChatMessage[]` carries each as its own message. During the moment between them, the "last assistant message" tail block IS a tool_use — the rich label appears. As the next assistant record arrives, `getActionInfo` re-evaluates against the new last message's last block. Any interleaved text or thinking block collapses the label.

**Bash Case 1 is green only because** Bash turns in a single-tool-per-turn shape terminate the turn on tool_result; there is no post-tool_use block to shift the tail. Read/Edit/Write cases chain more work, and any intermediate text/thinking lands on the tail.

**Why §10.3's divergence fix (§6.1.1) didn't address this:** §6.1.1 threaded `isWorkingOverride` so the composite `isWorking` can be true while `sessionStatus !== 'working'`. That lets `getActionInfo` be *called* in more cases (the early-gate at `ContextBar.tsx:342` no longer short-circuits to null). But calling `getActionInfo` is necessary, not sufficient — it must also return a rich label. When the tail block is text/thinking, it returns the wrong label regardless of whether `isWorking` is true.

**Candidate contributing factors, explicitly ruled in/out per E0's candidate list (§Item 1 bullet (a)–(d)):**

- (a) `useChat` snapshot churn — **ruled out as class driver.** Churn causes re-evaluation, not the wrong answer; the tail-block scan would always converge on the right tail if the tail were always a tool_use.
- (b) `hasUnmatchedToolUse` transient flip — **ruled out as class driver.** `hasUnmatchedToolUse` only gates `isWorking`; it does NOT choose the label. The label still comes from `getActionInfo`'s tail scan.
- (c) `isWorkingOverride` composite race — **ruled out as class driver per (a) reasoning.**
- (d) ChatPage re-render toggling — **ruled out as class driver per (b) reasoning.**

All four candidates affect *whether* a label is computed at all. The class bug is *which block* the label is computed from. The E0.3 regression to generic `"Working..."` is consistent with: `isWorking=true` (composite holds), `getActionInfo` runs, tail block is text → returns `'Composing response...'` → then `shouldSuppressComposingLabel` at `ContextBar.tsx:343–346` suppresses it because `lastActivityAt` is stale (tool's been running silently past STALE_ACTIVITY_MS=30s) → `jsonlAction=null` → `resolveActionLabel` returns null → `effectiveAction=null` → `getStatusInfo` Working branch returns `'Working...'` fallback at `ContextBar.tsx:110–116` pattern.

That path is consistent with the 74s-then-regress timing in E0.2/E0.3: the Read's rich label held while a tool_use was the tail; once Claude emitted a thinking or text block mid-stream, the scan caught the text tail, hit the staleness suppressor, and fell through to `'Working...'`.

**Contract-level fix shape.**

`getActionInfo` must be tool-aware, not tail-position-aware. Two shapes converge on the same end state:

- **Shape A (preferred, minimal).** Scan the last assistant message's content blocks in reverse and return the first tool_use branch match; fall back to thinking/text only if no tool_use is present in the message. This eliminates the tail-position dependency and matches Codeman's semantics (§6.6: "last write wins" on tool_use, not "last block wins").
- **Shape B (deeper, aligns with §6.2).** Reuse the `hasUnmatchedToolUse` tail scan's finding. When an unmatched tool_use exists anywhere in the visible tail, source the label from THAT tool's name (not from the last-assistant-last-block). This is the stronger contract — the label follows what's actually running, not what was most recently emitted — but requires either exposing the unmatched tool_use's `name`/`input` from `hasUnmatchedToolUse` (currently returns bool only) or having ContextBar run its own tail scan.

Shape B is what the §6 reference (Codeman) actually does — `transcript-watcher.ts` keeps `state.currentTool` as the name of the last UNMATCHED tool_use. Shape A is cheaper and fixes the common case; Shape B is correct in edge cases where Claude emits interleaved text mid-tool. Recommend Shape B for v5's code rotation if scope allows, Shape A otherwise — Shape A preserves the Bash win and covers 80%+ of Read/Edit/Write cases per the JSONL ordering above.

### 11.3 Item 3 — approval-modal DOM render gap (class: Class A, permanent variant)

**E0 citation:** Direct cause is the same upstream as Item 2. Separated as its own item because the failure mode is PERMANENT (browser refresh required) whereas Item 2 is EVENTUAL.

**Failure mode named: "subscription not active" (not "mount condition unmet" and not "state not reaching tree").**

Source evidence at `client/src/hooks/usePromptDetection.ts:66–69`:

```
useEffect(() => {
  if (!sessionId) return;
  const isActive = sessionStatus === 'working' || sessionStatus === 'waiting' || userJustSent;
  if (!isActive) return;
  …
}, [sessionId, sessionStatus, userJustSent]);
```

When `sessionStatus='idle'` and `userJustSent=false`, the effect returns without creating the polling interval. No subsequent `setInterval` fires. `res.prompts` never populates. `setPrompt` never calls. `<PermissionPrompt>` gate at `ChatPage.tsx:581` (`prompt && …`) never passes.

**Why a browser refresh fixes it:**

On refresh, `useChat` re-queries the session row; the server-side `sessions` table has the latest status computed by `status-poller` (which HAS flipped to 'waiting' server-side by the refresh point, either because the cooldown has expired or because the pane re-classified naturally). Fresh client state has `sessionStatus='waiting'` → `isActive=true` → poll starts → modal mounts. This confirms the bug is client-side subscription-gate, not server-side state.

**Cross-reference to Issue 15.1-F (subscription reinit gap):**

The dispatch asks if this echoes 15.1-F. It DOES — both are "WS event was emitted but client-side consumer was not listening at the moment of emit." The difference is the reconnection trigger: 15.1-F was about WS reconnection not replaying last-state; Item 3 is about an effect-dependency predicate gating the subscription itself. A fix for Item 3 does NOT automatically fix 15.1-F (different mechanism) but both share the class "subscription active iff predicate true, no last-known-state replay on predicate flip."

**Contract-level fix shape.**

Shape primary: Remove the `isActive` gate on `usePromptDetection`. Always poll when `sessionId` is set and the session row exists. The 2s steady-state cadence is cheap relative to the modal-correctness guarantee. Rationale: the gate was added (per the hook's own comment at `:65–66`) to avoid hammering tmux during idle sessions, but the cost of the gate (missed prompts) is higher than the cost it saves (a few tmux reads on idle sessions). Defense: scale the cadence to 5–10s when idle (the dispatch's C6 tests would pin the ≤3s bound only for active-status cases).

Shape secondary (converges with Item 2's secondary): server-push approvals via `session:prompt` WS event. The client then does not poll; it receives the prompt directly and mounts. Eliminates the class entirely.

**Does this also resolve 15.1-F?** No — 15.1-F is about last-state replay across WS reconnects. Addressing it separately remains queued.

**Tests-to-user-observable-contract (per C6):**

1. Given `session.status='idle'` on the client AND a prompt is present server-side, `<PermissionPrompt>` mounts within 3s of the initial page load (no refresh required).
2. Given a session that force-idled mid-approval-window, the modal eventually appears without a browser reload action.
3. Non-regression: idle sessions without prompts do NOT show a mounted PermissionPrompt (guard the fix against false-positive mounts).

**Two coupled causes, both confirmed by cited log lines.**

**Cause 1 — `isSessionWorking` uses the wrong coarse signal during the pre-tool thinking window.**

`ChatPage.tsx:329-332`:
```

  || (session?.status === 'working' && !heartbeatStale)

```

During Claude's pre-tool thinking window (after `userJustSent` clears ~40ms in, before `unmatchedToolUse` flips on JSONL tool_use arrival), the only live gate is `session.status === 'working'`. That flag is classified from tmux pane regex by the server status-poller. Case 2's capture shows `session.status` stayed `'idle'` for the full 20-second window while `sessionState.kind="Working:ToolExec"` was emitted simultaneously. `sessionState.kind` — the server's richer typed signal, already available at the client via `useSessionState` and already passed to ContextBar — is NOT consumed by `isSessionWorking`.

Cited log lines: Case 2 `B-entry` at T=+49ms, T=+9506ms, T=+15423ms, T=+19742ms all show `sessionStateKind:"Working", subtype:"ToolExec"`. Simultaneously `C-isSessionWorking` at T=+38ms through T=+15423ms show `sessionStatus:"idle", isSessionWorking:false`. `D-render` holds `"Idle — Waiting for instructions"` for 20 seconds.

`contextBarAction.ts:178-181`:

```
  return null;
```

This ignores `jsonlLabel`, `isWorking`, and every other input. When `sessionState.kind='Idle:*'` is STALE (server hasn't transitioned yet), the function clobbers the valid jsonlLabel that would have produced the rich per-tool label via the §6.1 Working-path inversion.

Cited log lines: Case 3 at T=+5948ms has `A-return {branch:"tool_use:Bash", value:{label:"Running command..."}}`; simultaneously `B-entry {isWorking:true, jsonlLabel:"Running command...", sessionStateKind:"Idle", subtype:"AwaitingFirstPrompt"}`; `D-pre-status {actionLabel:null}`; `D-post-status {statusInfoLabel:"Working..."}`; DOM showed generic `"Working..."` instead of rich `"Running command..."`.

**The two causes share one architectural asymmetry:** the client uses `session.status` as the sole Working/Idle GATE but uses `sessionState.kind` as the sole LABEL-SOURCE gate. These two server signals can disagree for 15–20 seconds at a time. The asymmetry produces two distinct failure modes depending on WHICH signal is stale.

### 12.4 Fix shape — contract level (no code)

Two complementary changes, both client-side, neither a revert of §6.1.1 / §6.4 / Item 3.

**Fix 1 — widen `isSessionWorking` to OR in typed `Working` kind.** At `ChatPage.tsx:329-332`, add `|| (sessionState?.kind === 'Working')` to the OR-chain. Closes the Case 2 gap: when `sessionState.kind='Working'` arrives before `session.status='working'`, the bar flips to working and begins consuming the rich label path.

**Staleness mitigation (load-bearing for the fix rotation):** naively ORing in `sessionState.kind='Working'` risks showing "working" during genuinely idle periods if the typed state is stale-Working (prior turn's state not yet transitioned). Mitigation candidates:
- Freshness predicate — gate the typed-state OR on `sessionStateUpdatedAt > lastUserMessageTs` (or equivalent per-turn marker), so stale pre-prompt Working is ignored after the user sends a new message.
- OR conjunction — require BOTH `sessionState.kind === 'Working'` AND (`unmatchedToolUse` OR `userJustSent` OR a post-send transcript-change signal).

The specific freshness gate belongs to the fix rotation's design. This §12 scope is to name that Fix 1 without staleness protection is insufficient.

**Fix 2 — in `resolveActionLabel`'s typed:Idle branch, fall through to legacy path when jsonlLabel is present AND isWorking is true.** Change `contextBarAction.ts:178-181` so that for `sessionState.kind === 'Idle'`:
- If `isWorking === true` AND `jsonlLabel` present → return `jsonlLabel` (extend the §6.1 Working-path inversion to cover stale-typed-Idle).
- Else → return null as today.

Preserves the true-idle render path (Idle → null → "Idle — Waiting for instructions") while refusing to let stale-Idle clobber live jsonlLabels. NOT a revert of §6.1; extends the inversion to a second sub-branch.

**Interaction between Fix 1 and Fix 2.** Fix 1 sets `isWorking=true` earlier by consuming `sessionState.kind='Working'`. Fix 2 uses `isWorking=true` to decide whether stale-typed-Idle can be overridden by jsonlLabel. Complementary: Fix 1 widens WHEN the bar considers itself working; Fix 2 widens WHICH label gets rendered while working. Together they close Case 2 and Case 3.

**Tests-to-user-observable contract (fix rotation will bake):**
- Given `session.status='idle'` but `sessionState.kind='Working'` AND a freshness gate permits it, ContextBar DOM reads a non-Idle label within 2s. (Case 2.)
- Given `sessionState.kind='Idle:AwaitingFirstPrompt'` (stale) AND `isWorking=true` (via any OR gate, including `userJustSent` for first 2-3s) AND `jsonlLabel='Running command...'`, DOM reads `"Running command..."` NOT `"Working..."`. (Case 3.)
- Non-regression: `sessionState.kind='Idle'` AND NO other Working signal AND jsonlLabel=null → DOM reads `"Idle — Waiting for instructions"`. (True-idle preserved.)
- Non-regression: stale-Working freshness gate does NOT false-positive idle sessions where the prior turn completed and typed state hasn't reset.

**Out of scope for §12.4 (acknowledged, not proposed):**
- Making the server's pane-regex classifier faster at flipping `session.status` in Case 2's pane shape — server-side Tier B+.
- Server-side transition of `sessionState.kind` out of stale-Idle faster — server-side.
- Removing the `isWorking` gate at `ContextBar.tsx:337` — gate is correct; fix is in gate inputs.
- Restoring Item 1's tail-run reverse-scan (commit `41c0f2c` reverted at `93b25dc`). Case 2's logs show Item 1's premise was **partial**: the label-collapse happens at `resolveActionLabel`'s typed-Idle branch (Cause 2), not at `getActionInfo`'s tail-block scan. In Case 2, `A-return` correctly found `branch:"tool_use:Read:memory"` during its brief window at +20008ms; the problem was that the bar was not WORKING at the time due to Cause 1, not that `getActionInfo` returned the wrong label. **Tail-run reverse-scan remains a reasonable hardening for multi-block assistant messages but is NOT the class root cause — it would not have closed Case 2 or Case 3.**
