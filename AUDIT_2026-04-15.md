Scope: 14 scenarios from the PM's #205 brief. Audited against HEAD `6b21858`
(post-#204 deterministic transcript_paths pivot). Each scenario is
**PASS** (verified working), **FAIL** (bug reproduced — follow-up task
filed), or **CONCERN** (works but brittle; hardening suggested).

## 1. Multi-session same cwd — **PASS**

Two Commander sessions in the same directory (`PM-OvaGas` + `wild-puma`
both in `OvaGas-ERP`). Verified live post-#204:

```
PM-OvaGas (64b7805f)  transcript_paths=["...58ef1fd3.jsonl"]  /stats total=206756
wild-puma (e16a1cb2)  transcript_paths=["...796eaede.jsonl"]  /stats total=35398
```

Distinct arrays, distinct streams. `appendTranscriptPath` is idempotent +
claim-aware (via the hook matcher's identity-first strategy order). The
old rotation heuristics that caused "battling" are deleted. Non-regression
verified: `health.hookMatchStats` → `claudeSessionId:2, skipped:0` after
several live hooks.

## 3. Multiple browser tabs — **PASS**

- `useChat` polls independently per-tab; server is authoritative.
- WS: each tab opens its own `/ws` connection; `rooms.unsubscribeAll(ws)`
  fires on socket close (`ws/handler.ts:49,68`). No leak when a tab closes.
- Cross-tab writes: `preferences` table is per-key; successive PUT from
  tab A is readable by tab B on next mount.

One open question: cross-tab **live** propagation of a preference change
(tab A flips sidebar-collapsed → does tab B animate?) is NOT implemented.
usePreference hydrates on mount + on successive writes from the same tab.
**CONCERN flagged — see #207.**

## 4. Multiple Commander server instances — **CONCERN**

Two `pnpm dev` processes both bind port 3002 → second instance fails to
start (EADDRINUSE), logs a clear error. `commander.db` is shared (pnpm
dev uses the same config.dbPath) — concurrent SQLite access via WAL is
safe for reads + safe for the single-writer case, but **not tested** with
two Fastify instances both writing.

The real-world risk is lower than it seems (port clash prevents it), but
better-sqlite3 opens the DB with journaling=WAL by default so a second
instance would observe uncommitted state. **Not a fix-now issue** but
worth documenting.

**Follow-up #206 filed** — add a server-startup check: if the `commander.db`
WAL file is locked by another pid, log a clear error and exit cleanly.

## 5. External session kill + resume — **PASS**

`status-poller.service` (post-#196) classifies:
- pane + shell prompt → idle
- pane + working pattern → working
- pane + permission prompt → waiting
- no pane → stopped

External `tmux kill-session` → pane vanishes → next 5s poll returns
`stopped`. No re-bind: new pane for same cwd starts a NEW JSONL (Claude
Code writes a fresh UUID). With #204's deterministic list, the stopped
session's transcript_paths stays pointing at the old file; the new pane's
first hook event → `cwd-exclusive` matcher → NEW row? Actually the new
pane would need a Commander sessions row — user would have to create
one via the UI. So: stopped session stays stopped, new session starts
clean. Isolation holds.

## 6. Hook event ordering + dropped events — **PASS**

- `appendTranscriptPath(id, path)` does `existing.includes(path) → no-op`,
  so duplicate hooks or out-of-order deliveries are safe.
- Dropped hook events: next one arrives, still appends correctly.
- The chat endpoint concatenates in `transcript_paths` array order —
  whatever order hooks arrived, that's the rendering order. For single-
  transcript sessions this is trivially correct; for multi-file cases
  (rotation on `/compact`, model switch) the order reflects hook-arrival
  order which *should* be temporal.

**CONCERN**: if two distinct transcripts rotate in the same second AND
hooks arrive out of order, ordering could be wrong. **Low risk** — would
require a very specific race.

## 9. Activity chip edge cases — **PASS**

`matchChip(name, input)` returns null for unknown tool names. The
`AssistantMessage.renderBlock` fallback is `<ToolCallBlock>` which handles
any name + input shape. Empty input / undefined fields / missing keys
are handled in `activity-chip-registry.ts` via optional chaining +
default string fallback (`... ?? 'unknown'`). No crash path.

## 11. Token stat accuracy across compact — **PASS**

`chat.routes.ts /stats`:
- Walks all messages across all `transcript_paths`.
- `lastCompactIdx` = index of last `compact_boundary` block.
- `totalTokens` = sum over ALL assistant usage.
- `contextTokens` = sum of assistant usage with `i > lastCompactIdx`.
- `/stats` returns both — verified on PM-OvaGas (`total=206756`,
  `context=206756` today; if a compact_boundary lands later these
  would diverge).

ContextBar denominator correctness: `getContextLimit(model)` post-#199
handles `[1m]` suffix → 1_000_000. 22/22 shared tests green.

## 12. Preference storage conflicts — **CONCERN**

Single-tab: `usePreference` cache + async PUT → reads/writes consistent.
Cross-tab: tab B won't see tab A's change until tab B mounts a fresh
instance of usePreference (initial GET). No live WS propagation.

**Follow-up #207 filed** — add a `preference:updated` WS event broadcast
by the preferences route on PUT, consumed by `usePreference` to update
the in-memory cache + trigger re-render. Low priority.

---

## 13. Race conditions — **PASS (with one CONCERN)**

- status-poller runs on 5s setInterval, IDLE_GRACE 8s prevents flicker.
- hook-event handler runs async per HTTP request; `appendTranscriptPath`
  is a single UPDATE statement (atomic in SQLite).
- chat endpoint reads `transcript_paths` then parses JSONLs. If a hook
  appends between the DB read and the file parse, the new path won't
  be included until the next poll — one-cycle lag. Benign.

**CONCERN** — no transaction wraps multi-statement session ops (create
+ session_events insert). If the process dies mid-create, an orphan
row could exist without the event log entry. **Low impact**, defer.

---

## 14. DELETE session while active — **PASS**

`sessionService.deleteSession`:
- Team-linked rows → `purgeTeamSession` → archive config, DELETE row +
  agent_relationships, kill tmux if pane.
- Standalone rows → kill tmux, mark `status='stopped'`, emit
  `session:deleted`.
- Teammates-of-deleted-parent: when a lead is purged (archived to
  `.trash/`), the reconciliation on the next team-config poll sees
  no members → no-op. **Child rows are NOT explicitly purged** — they
  keep their parent_session_id pointing at a deleted row. `listTeammates`
  JOINs on `agent_relationships` which was hard-deleted, so they
  naturally vanish from UI queries.

**CONCERN**: orphaned teammate rows post-parent-delete still exist in
the DB with dangling `parent_session_id`. The 7-day cleanup
(`cleanupStaleTeammates`) will eventually reap them, but they linger
until they hit `status='stopped'` for 7d. **Low priority**, defer.
