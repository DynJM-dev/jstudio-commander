## §2 — PATH forensics

### §2.1 — Where `claude` lives on Jose's machine

```bash
which claude  # in the active user shell
# → /Users/josemiguelbonilla/.nvm/versions/node/v22.17.0/bin/claude
```

Claude Code is installed as an npm global under NVM-managed Node 22.17.0. Not
under `/usr/bin`, `/opt/homebrew/bin`, `/usr/local/bin`. User-install location.

### §2.2 — PATH inherited by Finder-launched `.app`

macOS LaunchServices launches `.app` bundles with a minimal env:

```
PATH=/usr/bin:/bin:/usr/sbin:/sbin
```

This is the well-known "Finder launch has a different PATH than Terminal"
behavior — pre-login-shell defaults, no sourcing of `~/.zshrc` /
`~/.zprofile` / `~/.bashrc` etc.

### §2.3 — How the PATH propagates through the sidecar process tree

1. `jstudio-commander-shell` (Rust) starts with Finder's minimal PATH.
2. Tauri `app.shell().sidecar("sidecar-bin")` spawns the wrapper at
   `Commander.app/Contents/MacOS/sidecar-bin` inheriting Rust's env.
   The wrapper extends PATH only implicitly (N2.1 Task 2 Node-discovery
   loop finds `node` at standard install locations BUT does NOT export
   those directories onto PATH — it only assigns `NODE_BIN`).
3. Wrapper `exec "$NODE_BIN" …` replaces the shell with Node, inheriting
   the wrapper's (still-minimal) PATH.
4. Sidecar calls `node-pty.spawn('/bin/zsh', ['-i'], { env: process.env, … })`
   at `apps/sidecar/src/pty/manager.ts:79`. `process.env.PATH` is still the
   minimal Finder PATH.
5. `zsh -i` starts with `PATH=/usr/bin:/bin:/usr/sbin:/sbin`.
6. The N2.1-era default zdotdir (`sourceUserRc=false`) DOES NOT source
   `~/.zshrc` — so user's PATH exports never run. Intentional per
   ARCHITECTURE_SPEC (deterministic shell init, zero-latency OSC 133 A).
7. BootstrapLauncher writes `claude\n`. zsh tries `command -v claude` against
   `/usr/bin:/bin:/usr/sbin:/sbin`. No match. "command not found: claude".

### §2.4 — Why sourceUserRc=true is NOT the fix

The opt-in `preferences.zsh.source_user_rc=true` workaround would source
`~/.zshrc` and pick up user's PATH exports. BUT:

- Opt-in, requires Jose to flip a toggle — doesn't fix default behavior.
- Surfaces "fatal ~/.zshrc" / "slow ~/.zshrc" risks that N1 explicitly
  avoided (dispatch §1 Task 9 + PHASE_N1_REPORT §4 deviation).
- Still wouldn't work the first time a session is spawned on a clean install
  if the user never toggled it.

The right layer to fix is the sidecar-process-tree PATH, NOT the per-session
`.zshrc`. Analogous to the N2.1 Node-discovery fix location: the wrapper is
the single choke-point all downstream zsh sessions inherit from.

## §3 — Fix shape

### §3.1 — Layer

`native-v1/scripts/prepare-sidecar.sh` — the wrapper heredoc generated by
prepare-sidecar.sh that Tauri bundles at `Contents/MacOS/sidecar-bin`.

### §3.2 — Minimal fix

Extend PATH in the wrapper before `exec node`:

```bash
# Extend PATH with user install locations discovered during Node probe
# so node-pty-spawned zsh can resolve claude/node/npm/git/brew tooling.
EXTRA_PATHS=""
# Claude Code installer default:
[ -d "$HOME/.claude/local" ] && EXTRA_PATHS="$HOME/.claude/local:$EXTRA_PATHS"
# Node's bin dir (often contains npm-global `claude`):
[ -n "$NODE_BIN" ] && EXTRA_PATHS="$(dirname "$NODE_BIN"):$EXTRA_PATHS"
# Standard macOS user install locations:
for dir in /opt/homebrew/bin /usr/local/bin /opt/local/bin; do
  [ -d "$dir" ] && EXTRA_PATHS="$dir:$EXTRA_PATHS"
done
EXTRA_PATHS="${EXTRA_PATHS%:}"
[ -n "$EXTRA_PATHS" ] && export PATH="$EXTRA_PATHS:$PATH"
```

This runs AFTER the Node-discovery loop completes, so `NODE_BIN`'s directory
is definitively included.

### §3.3 — Why not fix in sidecar Node code

Alternatively, augment PATH in sidecar's node-pty spawn env at
`apps/sidecar/src/pty/manager.ts:70`. Rejected because:
- Same logical fix but one level too far downstream — if the sidecar
  itself needs external tools (git, etc. for future features), the wrapper
  is where they're needed too.
- The wrapper already owns the "find Node despite minimal PATH" problem;
  this is the same class of fix — the same code smell wants the same
  location.
- TypeScript sidecar vs. bash wrapper: the wrapper is where process-tree
  env composition happens; sidecar is business logic.
