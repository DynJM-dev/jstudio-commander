## §2 — The frontend focus chain, actual vs. intended

### §2.1 — Intended

1. Pane 2 is rendered with `focused={isFocused}` prop (WorkspaceLayout.tsx:91-98).
2. User clicks inside pane 2's terminal area.
3. SessionPane `onMouseDown={() => focusPane(index)}` (SessionPane.tsx:29) updates Zustand `workspaceStore.layout.focusedIndex = 1`.
4. WorkspaceLayout re-renders; pane 2's SessionPane gets `focused={true}`, pane 1 gets `focused={false}`.
5. Pane 2's xterm instance receives DOM focus. Its `term.onData` fires when user types.
6. Pane 1's xterm is blurred — doesn't receive keystrokes.

### §2.2 — Actual (as observed / code-read)

Steps 1-4 all work — `focused` prop propagates correctly via Zustand.
Step 5 **does not happen**. No code path calls `xterm.focus()` on the
newly-focused pane's Terminal instance.

**Proof by absence.** Exhaustive grep:

```bash
grep -rn "term\.focus\|terminal\.focus" native-v1/apps/frontend/src/
# → empty
```

Zero call sites. xterm.js's `Terminal.focus()` method is never invoked.

### §2.3 — What actually decides which pane gets keystrokes

Native DOM focus semantics. xterm.js creates an internal `<textarea>` on
`term.open(container)`. That textarea becomes a focusable DOM element.
When user clicks **inside** that textarea (or on xterm's helper canvas
that forwards clicks to it), the browser moves DOM focus to it and
keydown events fire its listener → `term.onData` fires.

If the user clicks **outside** the textarea (pane header, inter-pane
gutter, pane border, ContextBar, StateMdDrawer), DOM focus stays on
whatever had it before. In a freshly-loaded Commander window, that's
typically pane 1's xterm (the first one mounted + clicked).

**Bug reproduction shape:**
- Pane 1 spawns first, user clicks inside its terminal → pane 1's xterm
  textarea has DOM focus.
- User clicks `+ Pane` button → pane 2 spawns, `focusedIndex` updated
  to 1. But pane 1's xterm textarea still has DOM focus.
- User types — keystrokes route to pane 1's xterm → pane 1's session.
- Pane 2's border highlights ("focused" prop is true) but its xterm
  textarea isn't the DOM focus target.

From Jose's perspective: "I'm clearly in pane 2, but my input goes to
pane 1's chat input." Symptom matches exactly.

## §3 — Hypothesis mapping

Dispatch §2 Task 3 listed three hypotheses:

- **H1 — Focus state not propagating to xterm.js.** ✅ **CONFIRMED.**
  Zustand focusedIndex updates correctly but no `xterm.focus()` call
  bridges from Zustand state into xterm's DOM focus.
- **H2 — Global keydown listener capturing before xterm.js.** Not the
  cause. `WorkspaceLayout.tsx:41-53` adds a window keydown listener but
  it only captures Cmd+Opt+←/→ and Cmd+\; it doesn't capture general
  typing.
- **H3 — Chat textbox overlay.** Refuted — no chat textbox component
  exists in N2.1.x (grep confirmed in §1).

H1 is the full explanation.

## §4 — Fix shape (for Task 4)

Wire the `focused` state from SessionPane into TerminalPane as a prop,
and call `term.focus()` in a useEffect whenever `focused` becomes true.
Minimal surface change:

**SessionPane.tsx** — pass focused prop to TerminalPane:
```tsx
<TerminalPane key={sessionId} sessionId={sessionId} focused={focused} />
```

**TerminalPane.tsx** — new focused prop + useEffect bridge:
```tsx
interface Props {
  sessionId: string;
  focused?: boolean;
}

export function TerminalPane({ sessionId, focused }: Props) {
  // ... existing terminal setup ...

  // N2.1.4 Bug E fix: bridge Zustand focusedIndex → xterm DOM focus.
  // Without this, clicking pane 2's chrome only updates Zustand state; the
  // user's keystrokes stay routed to pane 1's xterm textarea (whichever had
  // DOM focus previously). See docs/diagnostics/N2.1.4-pane-input-routing-
  // evidence.md.
  useEffect(() => {
    if (focused && termRef.current) {
      termRef.current.focus();
    }
  }, [focused]);

  // ... rest unchanged
}
```

This bridges the existing Zustand-managed `focusedIndex` state into
xterm.js's DOM-focus machinery, so:
- Pane-click via SessionPane's onMouseDown → Zustand update → re-render
  with `focused=true` → useEffect fires → `term.focus()` → xterm textarea
  focused → keystrokes route to the right session.
- Cmd+Opt+←/→ focus cycle (WorkspaceLayout keydown handler) → Zustand
  update → same chain → focus flows with the keyboard shortcut too.
- Initial mount when `focused` starts as true: useEffect runs on mount,
  focuses the terminal. No-regression for pane 1's default focus.

**Effort:** ~10 min code (2 files, ~6 lines delta total).

**Dep hygiene (G12):** no new imports; uses existing xterm Terminal
instance + useEffect. `pnpm install --frozen-lockfile` N/A.

## §5 — Why this is at a different layer from Bug D

Bug D is sidecar-side: sidecar's `pty.stdin` write doesn't emit a byte
that Claude Code TUI interprets as "submit the paste". The fix is in
`apps/sidecar/src/pty/bootstrap.ts`.

Bug E is frontend-side: React's Zustand state representing "which pane
is logically focused" doesn't translate into browser DOM focus on the
right xterm textarea. The fix is in `apps/frontend/src/components/
SessionPane.tsx` + `TerminalPane.tsx`.

Different language (TypeScript sidecar Node vs TypeScript frontend
React), different runtime (sidecar stdio vs browser DOM), different
file, different test suite. No shared root cause. CTO's dispatch §0
assumption ("D and E are different layers") is correct — CODER
diagnosed them separately, writing evidence for each before any fix.
