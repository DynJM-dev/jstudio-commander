## §2 — What the code actually does today

`apps/sidecar/src/pty/bootstrap.ts` — `BootstrapLauncher.onQuiet()`
(lines 157-167):

```ts
private onQuiet(): void {
  if (this.state !== 'wait-for-claude-ready') return;
  if (this.opts.plan.kind === 'inject') {
    // Write content + newline. Claude Code treats this as first user input.
    this.opts.handle.write(this.opts.plan.content);
    if (!this.opts.plan.content.endsWith('\n')) this.opts.handle.write('\n');
    this.opts.onInjected?.();
  }
  if (this.readyTimer) clearTimeout(this.readyTimer);
  this.state = 'done';
}
```

Flow (confirmed by N2.1.3 deep probe at `/tmp/n2.1.3-deep-probe.mjs` +
today's probe at `/tmp/n2.1.4-bootstrap-probe.mjs`):

1. zsh fires OSC 133 A → BootstrapLauncher writes `claude\n` → Claude Code boots.
2. After 800 ms of output quiescence → `onQuiet()` fires.
3. onQuiet writes bootstrap file content (233-line PM markdown) + `\n`.
4. Claude Code TUI ingests the bytes; renders them as `[Pasted text #N +M lines]` placeholders.
5. **No submit fires. Content sits in Claude's paste buffer awaiting Enter.**
6. Jose types `hello`. Claude TUI appends `hello` to the pending paste.
7. Jose presses Enter → composite `<bootstrap>hello` submitted as one message.

The code comment on line 161 claims "Claude Code treats this as first user
input." That was the design assumption. Empirically wrong.

to the persona prompt.

### §3.3 — Interpretation

Claude Code's TUI (Ink-based) detects multi-line rapid pty.stdin input as
a paste. Paste content lands in Claude's input buffer. A plain `\n` at the
end of the paste is treated as an embedded newline in the paste content
(the paste's last line), NOT as a line-submit. Only `\r` — the byte the
Enter key physically sends on macOS — fires submit semantics.

Claude Code's paste detection mirrors xterm.js's bracketed-paste logic but
doesn't actually REQUIRE bracketed-paste escapes (we didn't send any; it
still detected paste). The paste-vs-typed distinction is shape + timing
based: a large block of bytes arriving on stdin with embedded `\n`s and
no explicit `\r` = paste.

## §4 — Hypothesis mapping

Dispatch §2 Task 1 listed four hypotheses. My finding vs. each:

- **H1 — Missing newline.** Code DOES add `\n` if absent. Not the cause.
  Refuted.
- **H2 — Bootstrap written before Claude ready.** The 800ms quiet-period
  correctly waits for Claude to stop outputting before writing. Bootstrap
  lands at the right moment (paste placeholders render in Claude's input
  line — proof Claude IS ready). Refuted.
- **H3 — Bootstrap written to zsh before claude invoked.** Refuted — the
  paste placeholders are Claude-code-TUI specific, proving Claude is
  running. zsh doesn't render `[Pasted text #N]`.
- **H4 — `\r\n` vs `\n` issue.** Closest hit. Not exactly a line-ending
  issue but specifically: **`\r` is what submits Claude Code's paste
  buffer; `\n` alone doesn't.** Refined fix shape in §5.

None of the dispatched hypotheses exactly described the mechanism. The
real mechanism is Claude Code's TUI paste-buffer-vs-typed-input
distinction, where only `\r` (Enter-key semantics) commits a paste.

## §5 — Fix shape (for Task 2)

In `apps/sidecar/src/pty/bootstrap.ts` `onQuiet()`, after writing content +
optional `\n`, schedule a small-delay `\r` write that commits the paste:

```ts
private onQuiet(): void {
  if (this.state !== 'wait-for-claude-ready') return;
  if (this.opts.plan.kind === 'inject') {
    this.opts.handle.write(this.opts.plan.content);
    if (!this.opts.plan.content.endsWith('\n')) this.opts.handle.write('\n');
    // N2.1.4 Bug D fix: Claude Code TUI buffers multi-line pty input as a
    // paste; \n alone does not submit. Send \r (Enter-key) after a short
    // settle delay so the TUI fully registers the paste before committing.
    this.submitTimer = setTimeout(() => {
      this.opts.handle.write('\r');
    }, SUBMIT_DELAY_MS);
    this.opts.onInjected?.();
  }
  if (this.readyTimer) clearTimeout(this.readyTimer);
  this.state = 'done';
}
```

Plus:
- Add `SUBMIT_DELAY_MS` constant (I'll use 200 ms — conservative; probe
  proved 10 s works so 200 ms has huge margin).
- Add `submitTimer: NodeJS.Timeout | null` field + clear it in `cancel()`.
- Add a test (existing `bootstrap.test.ts` can be extended — vitest fake
  timers are already used for the quiet-period tests).

**Effort:** ~15 min code + ~10 min test.

**Dep hygiene (G12):** no new imports; uses existing `setTimeout` /
`handle.write` surface. `pnpm install --frozen-lockfile` N/A.

## §6 — What this evidence does NOT cover

- Whether a simpler fix (replace `\n` with `\r` inline, no delay) works.
  Probe did not test that shape. Trade-off: simpler code, but no
  between-content-and-submit gap if Claude's TUI isn't done processing
  the paste. Chose the delayed-\r approach for safety.
- Whether Coder sessions (different bootstrap file, 186 lines vs 233)
  behave differently. Both use same BootstrapLauncher + same TUI, so no
