## §1 — Symptom (as reported by Jose, PHASE_N2.1.2_REPORT §3)

All three session types (PM / Coder / Raw) open a terminal with plain zsh.
Claude Code never starts. Typing `Hi` returns `zsh: command not found: Hi`.
Terminal shows the zsh error:

```
/Users/josemiguelbonilla/.jstudio-commander-v1/zdotdir/.zshrc:source:5: no such file or directory: [wrong path]
```

Because the OSC 133 hook source fails, no `A` marker ever fires on the first
prompt. `BootstrapLauncher.onOsc133` (`apps/sidecar/src/pty/bootstrap.ts:119`)
is gated on `event.marker !== 'A'` — no `A` → `claude\n` is never written to
the pty → Claude never boots → PM/Coder bootstrap never injects.

The downstream effect (missing Claude spawn, Step 10 PARTIAL + Step 11 FAIL in
Jose's smoke) is **one failure**, not two. Fixing the hook path fixes both.

## §3 — Sidecar `__dirname`-relative resolution (actual vs. required)

`apps/sidecar/src/pty/hook-path.ts` resolves the hook path like so
(compiled form identical, preserved line numbers):

```ts
const SRC_DIR = dirname(fileURLToPath(import.meta.url));        // line 18

export function resolveHookPath(): string {
  if (process.env.JSTUDIO_OSC133_HOOK_PATH) {                   // line 21
    return resolve(process.env.JSTUDIO_OSC133_HOOK_PATH);
  }
  // Dev layout: src/pty/ → ../../../resources/osc133-hook.sh
  const devPath = resolve(SRC_DIR, '..', '..', '..', '..', 'resources', 'osc133-hook.sh');  // line 25
  if (existsSync(devPath)) return devPath;
  // Fallback: same directory as the compiled binary (Task 10 bundling).
  const binAdjacent = resolve(SRC_DIR, '..', '..', 'resources', 'osc133-hook.sh');           // line 28
  return binAdjacent;
}
```

### §3.1 — In production Tauri bundle

Running from inside `.app`:

- `hook-path.js` lives at
  `Commander.app/Contents/Resources/sidecar/dist/pty/hook-path.js`.
- `SRC_DIR` = `Contents/Resources/sidecar/dist/pty/`.

**Line 25 (devPath, 4 levels up):**
- `pty/` → `dist/` → `sidecar/` → `Resources/` → `Contents/`
- + `resources/osc133-hook.sh` → `Contents/resources/osc133-hook.sh`
- On macOS APFS (case-insensitive-case-preserving default), this would
  match `Contents/Resources/osc133-hook.sh` — **no file exists directly
  under Resources/** (the hook is one level deeper at
  `Contents/Resources/resources/osc133-hook.sh`). `existsSync` → **false**.

**Line 28 (binAdjacent fallback, 2 levels up):**
- `pty/` → `dist/` → `sidecar/`
- + `resources/osc133-hook.sh` → `Contents/Resources/sidecar/resources/osc133-hook.sh`
- **Does not exist** — `sidecar/` in the bundle contains only
  `dist/`, `node_modules/`, `package.json`. Verified via `ls` in §2.

So `resolveHookPath()` returns the wrong path. `ensureZdotdir()` writes a
`.zshrc` that `source`s that wrong path. zsh fails at line 5 of the
generated `.zshrc` with `no such file or directory` — matching Jose's
terminal evidence verbatim.

### §3.2 — Why dev mode works (for the record)

In dev via `pnpm sidecar:dev` (tsx watching `src/`):
- `SRC_DIR` = `native-v1/apps/sidecar/src/pty/`.
- Line 25: 4 levels up = `native-v1/`. + `resources/osc133-hook.sh` =
  `native-v1/resources/osc133-hook.sh` — **exists** (authored repo path).
  So dev hits the devPath branch and works.

Dev never exercises the fallback branch. The fallback shape was untested
against production-bundle reality — classic N1-close smoke gap per
SMOKE_DISCIPLINE.md §0.

### §3.3 — Why PM's hypothesis is correct

Dispatch §3 Task 1 hypothesis (verbatim): *"sidecar __dirname-relative
vs Tauri bundle.resources top-level; one level too deep — extra `sidecar/`
segment."* Confirmed: fallback resolves to
`Contents/Resources/sidecar/resources/osc133-hook.sh` while bundle places
the hook at `Contents/Resources/resources/osc133-hook.sh`. The extra
`sidecar/` segment is exactly what PM identified via direct `ls`.

## §4 — Available override hook

`hook-path.ts:21` already honors `JSTUDIO_OSC133_HOOK_PATH` before any
filesystem probing:

```ts
if (process.env.JSTUDIO_OSC133_HOOK_PATH) {
  return resolve(process.env.JSTUDIO_OSC133_HOOK_PATH);
}
```

This means **Option A (Rust-owned PathResolver → env var → sidecar) requires
zero sidecar code changes.** The sidecar side is already covered by the
existing override. The fix sits entirely in the Rust shell at spawn time,
which is where the Tauri `PathResolver` API lives anyway.

## §5 — Fix shape (preview for Task 2, per dispatch §3 Task 2)

**Option A — chosen.** In `apps/shell/src-tauri/src/lib.rs` `spawn_sidecar()`,
call `app.path().resource_dir()` (Tauri v2 `PathResolver` API), join
`resources/osc133-hook.sh`, and set it on the sidecar child's environment
via `tauri_plugin_shell::Command::env("JSTUDIO_OSC133_HOOK_PATH", …)`.
Also expose the resolved resource_dir as `JSTUDIO_RESOURCE_DIR` for any
future Tauri-bundled resource (cost: one additional `.env()` call).

**Why not Option B (adjust bundle.resources to put hook under sidecar/).**
Viable but couples bundle layout to sidecar internals — every future
bundled resource (icons, templates, defaults) would have to co-locate
under `sidecar/`. Option A lets Rust own resource resolution exactly
once and exports the answer to the sidecar; the pattern generalizes to
any resource.

**Rust LOC impact.** ~10-15 lines added, well within 138/150 budget.

**Dep hygiene (G12).** `app.path()` is stock Tauri v2 (`tauri::Manager`
is already imported at lib.rs:11). No new crate, no package.json churn.
