# N2.1.5 — Bug D residual evidence: cold-launch autosend race

**Phase:** N2.1.5
**Task:** Task 1a — diagnose Bug D residual (intermittent first-cold-launch autosend failure).
**Layer (SMOKE_DISCIPLINE §2):** **sidecar pty-write timing layer.** Extension of the N2.1.4 pty.stdin/shape layer. The `\r` commit byte is still correct; what's wrong is the *timing relationship* between the content write and the `\r` write when Claude Code's cold-boot output pattern has large gaps.
**Authored:** 2026-04-22, before Task 2 fix (G10).

## §1 — Symptom (Jose's N2.1.4 smoke §3 Step 10)

> "First attempt: Submit spawned the session but bootstrap did NOT auto-send. Jose typed 'HI' and Claude received `<bootstrap text>HI` concatenated. Reliably reproducible per Jose ("bootstrap is actually working for now" on

## §2 — Reproduction (CODER probe)

### §2.1 — Probe setup

`/tmp/n2.1.5-cold-probe.mjs` — three sequential PM-session spawns against
a freshly Finder-launched Commander.app (all prior processes killed;
runtime state cleared). Each probe:

1. Opens a WS subscription to `global` + `session:<id>`.
2. POSTs `/api/sessions {sessionTypeId:'pm',…}`.
3. Captures pty output with chunk-level timing for 20 s.
4. Scores `bootstrapTextHits` (tokens Claude rendered after paste commit)
   and `pastePlaceholders` (unsubmitted paste badges).

```
           …  (8-second silence — NO output)

```

**Key observation:** Claude's pty went SILENT from t+1378ms through
t+9357ms. The existing `onQuiet` quiet-period is 800 ms, so it fired
at ~t+2178ms. Bootstrap content was written to pty.stdin at that moment,
plus a `\r` at ~t+2378ms (200 ms later). **Claude did not respond to
either write** during the 8-second silence window.

Compared to cold-1 (success case), the last pre-silence chunk was at
t+2189ms with +828B — chunks kept flowing THROUGH the onQuiet boundary,
giving Claude's TUI enough continuous activity to process the paste when
it landed.

### §2.4 — Root-cause hypothesis

**The 800 ms quiet-period heuristic is racy on cold launch.** When
Claude's cold-boot has a brief (> 800 ms) silence gap BEFORE its paste
handler is fully registered, `onQuiet` fires → content + `\r` land in a
half-initialized state → content effectively lost (no response).

The 200 ms submit delay can't save us here — it measures time between
our writes, not Claude's readiness. The race is:

```
t+1378ms: Claude finishes boot chrome burst, goes silent
          (maybe waiting on filesystem cache, config read, MCP socket)
t+2178ms: quiet-period fires (800ms since last output)
          → write content + \n
t+2378ms: submitTimer fires (200ms after content write)
          → write \r
          → Claude receives but cannot process (not yet ready)
t+9357ms: Claude comes back online — too late; \r already consumed as
          empty submit or stranded in a paste buffer that never commits
```

## §3 — Why warm / subsequent launches work

On warm launches, Claude Code's node process pages are already hot (JIT
caches, module caches, filesystem metadata). Cold-boot chrome rendering
streams continuously without > 800 ms gaps, so `onQuiet` fires AFTER
Claude's TUI is fully paste-ready. Content + `\r` land cleanly.

My probe confirms this: cold-1 and cold-2 BOTH succeeded, but they
were in fact warm relative to cold-3 (sidecar had spawned 2 previous
sessions whose processes already primed filesystem + memory caches).
Cold-3 was a fresh pty spawn; even within the "3 sequential spawns"
test, environmental variance triggered the gap.

## §4 — Fix shape (for Task 2)

Replace the fixed 200 ms submit delay with a **post-content-write quiet
detection loop** that waits for Claude's pty output to stabilize AFTER we
write content, bounded by a hard deadline:

```ts
// New state: 'wait-for-paste-quiet' — after we write the content, reset
// a timer on every new output chunk. When output stays quiet for
// submitDelayMs, OR we hit submitMaxWaitMs, write \r.
private onQuiet(): void {
  if (this.state !== 'wait-for-claude-ready') return;
  if (this.opts.plan.kind === 'inject') {
    // Wrap in bracketed paste markers. Explicit "this is a paste" signal
    // to Claude regardless of timing heuristics — the terminator
    // \x1b[201~ tells Claude the paste ends, so the subsequent \r is
    // unambiguously a submit.
    this.opts.handle.write('\x1b[200~');
    this.opts.handle.write(this.opts.plan.content);
    if (!this.opts.plan.content.endsWith('\n')) this.opts.handle.write('\n');
    this.opts.handle.write('\x1b[201~');
    this.state = 'wait-for-paste-quiet';
    this.submitHardDeadline = Date.now() + this.submitMaxWaitMs;
    this.submitTimer = setTimeout(() => this.commit(), this.submitDelayMs);
    return;
  }
  if (this.readyTimer) clearTimeout(this.readyTimer);
  this.state = 'done';
}

onData(chunk: string): void {
  if (this.state === 'wait-for-claude-ready') {
    // existing logic
  } else if (this.state === 'wait-for-paste-quiet') {
    if (this.submitTimer) clearTimeout(this.submitTimer);
    if (Date.now() > this.submitHardDeadline) {
      this.commit();
      return;
    }
    this.submitTimer = setTimeout(() => this.commit(), this.submitDelayMs);
  }
}

private commit(): void {
  if (this.state !== 'wait-for-paste-quiet') return;
  this.opts.handle.write('\r');
  this.state = 'done';
  this.submitTimer = null;
  if (this.readyTimer) clearTimeout(this.readyTimer);
  this.opts.onInjected?.();
}
```

**Two changes combined:**

1. **Bracketed paste wrappers (`\x1b[200~…\x1b[201~`)** — explicit
   protocol signal to Claude's TUI that this byte range is a paste,
   independent of chunk-size heuristics. Works even if the TUI hasn't
   enabled its internal paste detection yet. Terminator then marks
   "paste done" → subsequent `\r` is unambiguously a submit.
2. **Post-write quiet detection with hard deadline** — replaces fixed
   200 ms timer. Waits for `submitDelayMs` of quiet AFTER writing
   content, up to `submitMaxWaitMs` hard deadline (default 3000 ms).
   On typical launches: commits quickly (~submitDelayMs after Claude's
   last response chunk). On slow cold launches: commits at hard
   deadline rather than racing. On the pathological 9-second silence
   case: commits at deadline; Claude picks it up when it comes back.

**Defaults:** `submitDelayMs = 300`, `submitMaxWaitMs = 3000`.

**Effort:** ~30 min code + ~20 min test (need fake timers for the
two-stage wait).

## §6 — What this evidence does NOT cover

- **Whether bracketed-paste wrappers work on Raw sessions.** Raw's
  plan.kind is 'skip' so no content/wrapping is written. No expected
  impact. Verify via post-fix probe.
- **Whether the Claude TUI 9-second silence case is actually a stuck
  TUI or a slow MCP/config handshake.** Root cause of WHY Claude goes
  silent for 9 seconds is outside the sidecar's control; our fix makes
  the commit robust against those gaps, not the gaps themselves.
- **Upstream Claude Code behavior.** If Claude Code's TUI changes its
  paste handling in a future version, we may need to revisit. The
  bracketed-paste wrapper is stable protocol (xterm conventions since
  mid-2000s) — low churn risk.
