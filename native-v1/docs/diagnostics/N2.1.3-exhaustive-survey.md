**§1.2a — Probe design note on OSC 133 markers:**

My direct probe to send `sleep 2 && echo done\n` to a Raw session and observe
B/D markers FAILED to observe any. Root cause: Raw sessions ALSO launch
`claude` (session-types seed: `raw.clientBinary='claude'`,
`raw.bootstrapPath=null`). BootstrapLauncher writes `claude\n` on first A
marker; subsequent shell commands go to claude's input buffer, not zsh's.
This is probe design, not infrastructure: OSC B/D coverage is proven by
orchestrator.test.ts. A properly isolated probe would need
`skipClientLaunch:true` (internal test flag, not exposed to UI).

## §2 — FINDINGS (items that DID surface real gaps — not fixed in N2.1.3)

### §2.1 — FINDING 1: PUT /api/preferences rejects non-string values (SQLite binding error)

**Severity:** MEDIUM (latent; frontend currently sends strings, doesn't
trigger; any future client sending a boolean/number crashes).

**Layer:** sidecar HTTP → drizzle → better-sqlite3 binding.

**Evidence:**

```bash
curl -X PUT http://127.0.0.1:11002/api/preferences/zsh.source_user_rc \
  -H 'content-type: application/json' -d '{"value":true}'
# → 500 {"message":"SQLite3 can only bind numbers, strings, bigints, buffers, and null"}

curl -X PUT http://127.0.0.1:11002/api/preferences/zsh.source_user_rc \
  -H 'content-type: application/json' -d '{"value":"true"}'
# → 200 {"ok":true,"key":"zsh.source_user_rc"}
```

**Root cause hypothesis:** `preferences.value` column is TEXT per the
Drizzle schema; the PUT handler writes `req.body.value` directly without
stringifying. Boolean/number bodies → `bindValue(true)` → SQLite binding
error → 500.

**Fix shape (NOT IN SCOPE FOR N2.1.3):** single-line `String(req.body.value)`
coercion + document typed preference values in the API contract. Or: add a
runtime schema validator (zod) to every PUT route. ~10 lines.

**Routing:** §7 Tech debt row. Not blocking any current feature (frontend
sends strings). Fold into N3 scope review OR a future hardening pass.

### §2.2 — FINDING 2: Event shape mismatch between docs and implementation (session:state)

**Severity:** LOW (probe artifact, but a real readability concern).

**Layer:** shared WsEvent types.

**Evidence:** My initial probe filtered `ev.transition` + `ev.status` on
session:state events — both undefined. Actual shape per orchestrator.ts:278
is `{type:'session:state', sessionId, state:{kind, since|commandStartedAt|
exitCode|...}, timestamp}`. The `state.kind` nesting is correct per
packages/shared SessionState type union, but frontend code and debug
probes commonly want `state` flat. Documentation gap.

**Fix shape (NOT IN SCOPE):** add a helper `eventSummary(ev)` in shared/
that flattens state.kind for display. Or: a one-line JSDoc comment on
WsEvent.SessionState making the nested shape explicit.

**Routing:** §7 tech debt — documentation polish.

### §2.3 — FINDING 3: `BootstrapLauncher` runs `claude` for ALL session types including Raw

**Severity:** LOW (design observation, not a bug).

**Layer:** orchestrator.ts:194 + bootstrap.ts:127.

**Observation:** The session-types seed gives Raw `clientBinary='claude'`
(same as PM/Coder) with `bootstrapPath=null`. BootstrapLauncher writes
`clientBinary\n` on first OSC 133 A marker regardless of plan.kind; Raw
sessions therefore run claude without bootstrap content injected.

**Is this intentional?** The comment at orchestrator.ts:194 reads "for
PM/Coder/Raw" — yes, intentional. But the UX contract is ambiguous: a
user spawning "Raw" might expect plain zsh, not a default Claude session
without persona.

**Routing:** §8 Question — confirm PM/CTO intent on Raw semantics. If
intent is "zsh only", `clientBinary=null` on Raw session-type row is the
fix. If intent is "Claude without persona" (current behavior), worth
clarifying in the session-types documentation.
