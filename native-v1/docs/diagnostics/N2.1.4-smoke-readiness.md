## §3 — Bug E fix: CODER verification limits

CODER cannot click UI panes. Verification of pane-focus routing requires
user click-through. Limits of smoke-readiness for Bug E per
SMOKE_DISCIPLINE.md §3.4:

- **Code-read confirmation:** `grep -rn "term\\.focus\\|terminal\\.focus"
  native-v1/apps/frontend/src/` now returns exactly one match, in
  `TerminalPane.tsx`'s new useEffect. Previously zero matches — that's
  the root cause addressed by the fix.
- **Prop threading:** SessionPane passes `focused={focused}` to
  TerminalPane; TerminalPane's `useEffect` with `[focused]` dep + guard
  against null termRef is idiomatic React.
- **Idempotency:** xterm.js `Terminal.focus()` on an already-focused
  instance is a no-op (confirmed in xterm.js source — guarded by
  `this._core.textarea.focus()` with native DOM focus semantics).
- **Initial mount:** pane 0 defaults to `focused=true`; useEffect fires
  on mount → pane 0 gets focus. No regression vs prior behavior.

**What Jose verifies that CODER cannot:**
- Pixel-level confirmation that clicking pane 2 actually lets keystrokes
  flow to pane 2's session.
- Cmd+Opt+←/→ now routes keystrokes (not just moves the focus border).
- No xterm focus-stealing pathology at 3 concurrent panes.
