## §1 — Symptom (as reported by Jose's N2.1 re-dogfood)

Finder-launched `Commander.app` from `pnpm build:app` release artifact. Preferences modal (Cmd+,) displays:

> Sidecar unreachable — tried 127.0.0.1:11002..11011. Ensure the sidecar process is running (Rust shell auto-spawns it in prod; in dev run 'pnpm sidecar:dev').

This message is emitted by `apps/frontend/src/lib/sidecarUrl.ts` lines 41–50 when `runDiscovery()` exhausts every port without seeing a `/api/health` 200.

## §2 — Layer of the failure (per SMOKE_DISCIPLINE.md §2)

The smoke stack:

```
USER → FINDER → .APP → TAURI V2 (WKWebView) → REACT → WEBVIEW FETCH → SIDECAR HTTP → DRIZZLE
```

Under the symptom above:
- **Sidecar HTTP layer: OK.** `ps aux` shows `/usr/local/bin/node .../Commander.app/Contents/Resources/sidecar/dist/index.js` spawned by `jstudio-commander-shell`. `runtime.json` is written with `{port:11002, pid:…}`. `curl http://127.0.0.1:11002/api/health` returns `{"status":"ok","version":"0.1.0-n1","uptime":…}` in <5ms. N2.1 verified this path.
- **Webview-fetch layer: FAILING.** The frontend's `fetch('http://127.0.0.1:11002/api/health', {signal})` never resolves with a 200 response from inside the webview's JS context.

The N2.1 hotfix verified the Sidecar HTTP layer (correct for that hotfix's scope — the Rust-side sidecar spawn path) but did not exercise the webview-fetch layer. Per SMOKE_DISCIPLINE.md §4.2 this is the anti-pattern that motivated the standard's creation.

## §3 — Root-cause hypothesis + evidence

This hotfix operates without interactive webview DevTools access (CODER running in a scripted continuing spawn). Evidence is assembled from:

1. Reading the current Tauri v2 config (`apps/shell/src-tauri/tauri.conf.json`) — confirmed.
2. Reading the frontend fetch code (`apps/frontend/src/lib/sidecarUrl.ts`) — confirmed.
3. Reading the sidecar server middleware stack (`apps/sidecar/src/server.ts`) — confirmed.
4. Tauri v2 CSP docs (https://v2.tauri.app/security/csp/).
5. WKWebView Cross-Origin Resource Sharing + App Transport Security references.

### 3.1 — Evidence A: Tauri `csp: null` + a cross-origin fetch

`tauri.conf.json` holds `app.security.csp: null`. The frontend is served from Tauri's custom `tauri://localhost` (or `http://tauri.localhost` on some platforms) origin. The fetch target is `http://127.0.0.1:11002` — a different scheme AND different port AND different host (`localhost` ≠ `127.0.0.1`), so triply cross-origin.

Per Tauri v2 CSP docs: when `csp: null`, Tauri does not inject a meta CSP tag, but on macOS WKWebView applies its own same-origin policy + ATS regardless of Tauri. A fetch from `tauri://localhost` to `http://127.0.0.1:11002` is subject to:

- **CORS** — the response must include `Access-Control-Allow-Origin` for the JS caller to read it. Without this header the browser blocks the response body and `fetch()` rejects.
- **ATS (App Transport Security)** — HTTPS-like origins (including `tauri://` treated as secure by WKWebView) calling `http://…` may be blocked outright as insecure. Localhost is generally exempted on macOS 10.14+ but Tauri apps inherit whatever ATS policy the generated Info.plist ships (currently no `NSAppTransportSecurity` keys, so OS defaults apply).

### 3.2 — Evidence B: Sidecar has no CORS headers

The sidecar Fastify server (`apps/sidecar/src/server.ts`) registers:
- `fastifyWebsocket`
- `healthRoutes`
- `sessionTypesRoutes`
- `preferencesRoutes`
- `sessionRoutes`
- `scrollbackRoutes`
- `projectRoutes`
- `workspaceRoutes`

**There is no CORS middleware** (no `@fastify/cors`, no manual `Access-Control-Allow-Origin` header setter). Responses to webview fetches therefore omit CORS headers entirely.

Grep evidence:

```
$ grep -n "cors\|Access-Control" apps/sidecar/src/**/*.ts
(no matches)
```

This alone is sufficient to block any cross-origin fetch from the webview, regardless of whether Tauri adds a CSP restriction.

### 3.3 — Evidence C: the frontend fetch uses native `fetch()`, not `@tauri-apps/plugin-http`

`apps/frontend/src/lib/sidecarUrl.ts:57` calls `fetch(…)` — the browser-native WKWebView fetch, which goes through all of the above enforcement. `plugin-http` (if used) would route through Rust's reqwest and bypass CORS/ATS. We are not using it.

### 3.4 — Ruled out

- **Sidecar not running:** ruled out (`ps aux` confirms).
- **Wrong port:** ruled out (`runtime.json` confirms 11002, range-scan handles drift).
- **Node not in PATH:** already fixed in N2.1 (wrapper discovers Node across standard install locations).
- **Gatekeeper quarantine on wrapper:** ruled out (`xattr` shows only `com.apple.provenance`).

### 3.5 — Primary hypothesis

**Root Cause B — WKWebView cross-origin fetch blocked by missing CORS headers on the sidecar.**

The webview sends the request (the sidecar's Fastify log will show it arriving), but the response lacks `Access-Control-Allow-Origin`, so WKWebView's JS layer sees a network error and `fetch()` rejects. The rejection is caught by `pingHealth()`'s `try/catch`, which returns `false`, and after all ports are probed `runDiscovery()` throws the "Sidecar unreachable" message.

This matches the reported symptom exactly and is consistent with all three pieces of code evidence above.

### 3.6 — Secondary hypothesis (additive)

**Root Cause A — Tauri v2 production CSP default.** Even with `csp: null` in config, certain Tauri v2 release-mode behaviors may still influence `connect-src`. To be safe, this hotfix explicitly sets a CSP that allows `connect-src http://127.0.0.1:* ws://127.0.0.1:*` so the defense-in-depth case is covered.

## §4 — Fix strategy (informs Task 2, not applied in this evidence file)

Preference ordering per dispatch §4 Task 2:

1. **CORS on sidecar** (required) — register `@fastify/cors` with a permissive policy scoped to localhost origins. Fixes the browser-blocked-response case definitively.
2. **Explicit CSP in tauri.conf.json** (required as defense in depth) — `connect-src 'self' http://127.0.0.1:* ws://127.0.0.1:*` plus standard directives. Fixes the Tauri-CSP-default case if it's contributing.
3. **Plugin-http migration** (deferred, unless above fail) — larger scope.

Both #1 and #2 are low-scope, fast to apply, and independently valuable. The Task 2 commit applies both and notes in its body which of the two was the necessary cause vs. belt-and-suspenders — determined by whether Jose's DevTools Network tab shows the request completing with CORS headers (necessary cause #1) or a CSP violation (necessary cause #2).

## §5 — Why full interactive DevTools inspection is deferred to Jose's user-facing smoke

Per SMOKE_DISCIPLINE.md §3.4: CODER cannot self-certify user-facing smoke. The DevTools inspection that provides pixel-level Network/Console evidence is part of the smoke Jose runs — CODER's role is to (a) identify root-cause hypothesis from code evidence (this document), (b) ship the fix + `pnpm build:app:debug` DevTools-enabled build (Task 4), (c) confirm smoke-readiness, (d) let Jose's interactive DevTools session provide the final ground-truth observation.

Jose's user-facing smoke — specifically dispatch §3.1 step "confirm the Network tab shows the frontend's /api/health request completing with HTTP 200 status" — is where the hypothesized root cause is empirically confirmed or refuted. If the Network tab shows CORS error, hypothesis is correct and the fix shipped in Task 2 resolves. If it shows CSP violation only, Evidence B fix was sufficient without CORS. If neither, we escalate to N2.1.2 with the DevTools output as new evidence.
