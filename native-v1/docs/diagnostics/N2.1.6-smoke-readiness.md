## §3 — Bug K: scrollback UTF-8 round-trip verification

Direct probe at `/tmp/n2.1.6-verify-kill-k.mjs`:

```
writer raw: "Opus 4.7 (1M context) │ ctx — │ $0.00\n⏵⏵ auto mode on\n
            ◉ xhigh · /effort\n──────────────────\n✳ Claude Code"
```

Content includes every mojibake-prone glyph class: em-dash (U+2014),
bullet (U+2022 / U+25C9), ⏵⏵ arrow (U+23F5), ──────── box-drawing
(U+2500), ✳ asterisk (U+2733), │ vertical line (U+2502).

| Verification | Result |
|---|---|
| Writer base64 → sidecar PATCH → sidecar Buffer → BLOB | 200 OK, 155 bytes stored |
| Sidecar GET scrollback → base64 round-trip | raw string match ✓ |
| OLD path `atob(blob)` → binary string contains `â` (U+00E2) | ✓ mojibake confirmed present in old path |
| NEW path `Uint8Array.from + TextDecoder('utf-8')` | original string recovered ✓ |
| `term.write(Uint8Array)` uses xterm's internal UTF-8 decoder | same correct recovery (xtermjs PR #1904) |

The OLD path's mojibake signature is exactly what Jose observed: UTF-8
byte `0xE2` (starts em-dash, bullet, box-drawing, ⏵, ✳ sequences)
rendered as U+00E2 = `â`. The NEW path correctly decodes these bytes
back to their original Unicode codepoints.
