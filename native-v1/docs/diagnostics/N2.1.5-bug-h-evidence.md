`apps/frontend/src/components/TerminalPane.tsx` lines 40-68 (current
verbatim, with post-N2.1.4 focused-prop additions):


Container styling:

```css

```

Flex child inside flex parent with `minHeight: 0` — a layout pattern
that frequently produces 0-dimension or pre-layout `getBoundingClientRect`
reads on the first microtask after commit.

## §3 — Mount timing analysis

1. React commits TerminalPane → DOM created, container element exists.
2. useEffect fires SAME TICK → `term.open(container)`, `WebglAddon.load`,
   `fit.fit()` all run before browser has performed the first layout pass
   on the flex chain.
3. `fit.fit()` calls `FitAddon.proposeDimensions()` which reads
   `container.getBoundingClientRect()`. On a flex-1 child of a flex-1
   parent during initial commit, this frequently returns pre-layout
   values (reported width `< 10` or rounded-down pixel fractions).
4. xterm.js calls `term.resize(cols, rows)` with the bad proposal. WebGL
   renderer caches its cell atlas against those miscomputed cell
   dimensions.
5. Async scrollback fetch (100-400 ms later) calls `term.write(decoded)`.
   Content renders into the bad atlas — producing Jose's exact observed
   shape: **text overlaps, wrong line breaks, garbled content.**
6. User's first keystroke → `term.onData` → pty echo → `term.write()`
   triggers renderer dirty-flag → implicit re-measure/re-render against
   now-stable DOM dimensions → atlas rebuilt → artifacts disappear.

Step 6 matches Jose's "kinda gets fixed when I write on it" verbatim.

## §5 — Fix shape (for Task 3)

Minimal, one-line change in `TerminalPane.tsx` L68. Replace the
synchronous `fit.fit()` call with a requestAnimationFrame-deferred one
so the browser's layout pass completes before we measure container
dimensions:

```ts
// Before (current)
fit.fit();

// After (N2.1.5 Bug H fix)
// Defer fit to the next animation frame so the flex container has
// completed layout before we measure its dimensions. Synchronous
// fit.fit() on the same tick as term.open() racing React's commit
// in a flex-1-in-flex-1 container is the N2.1.5 Bug H root cause —
// xterm.js #5320 / #4841 / #3584 describe the exact pattern.
requestAnimationFrame(() => {
  try { fit.fit(); } catch { /* container still sizing */ }
});
```

Already-present `onResize` handler (L70-76) uses the same try/catch
pattern — this is consistent idiom.

**No dep changes.** No new imports. G12 N/A.

**Effort:** ~3 min code change + ~5 min rebuild + smoke-readiness
verify.

## §7 — What this evidence does NOT cover

- **Whether WebGL addon should also be delayed.** Currently loaded
  before `fit.fit()`. The fix only moves `fit.fit()` to the next frame;
  WebGL stays on the current tick. If artifacts persist after the
  `requestAnimationFrame` fix, next step is to move `term.loadAddon(webgl)`
  into the same deferred callback. Starting minimal per dispatch
  guardrail.
- **Whether a regression could affect WebGL performance.** The deferred
  `fit()` runs one frame (~16 ms) after `term.open()`. User-perceivable
  lag: none; terminal mount is still < 100 ms end-to-end.
- **Whether scrollback restoration needs to wait for `fit()`.** Current
  code reads scrollback via async fetch THEN calls `term.write()`. The
  fetch takes > 16 ms typically, so by the time `term.write()` runs,
  the deferred `fit.fit()` has already executed. No additional
  ordering change needed.
