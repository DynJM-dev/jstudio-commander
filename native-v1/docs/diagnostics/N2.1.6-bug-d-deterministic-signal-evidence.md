# N2.1.6 — Bug D evidence: deterministic Claude-ready signal identification

**Phase:** N2.1.6
**Task:** Task 1 — find and verify a deterministic signal that fires only
when Claude Code's TUI is ready to receive paste input. Replace the 3-s
hard-deadline in N2.1.5's `wait-for-paste-quiet` state with this signal.
**Layer (SMOKE_DISCIPLINE §2):** **sidecar pty-output parse layer.** Same
layer as prior Bug D rotations; mechanism evolved from chunk-gap timing
→ byte-pattern signal detection.
**Authored:** 2026-04-22, before Task 2 fix (G10).

## §1 — Symptom (Jose N2.1.5 smoke §3 Step 10)

> "FAIL on first cold-launch — PARTIAL FIX ONLY. First-attempt cold-launch:
> bootstrap did NOT fire. Jose had to press Enter manually. Subsequent
> session opened within same app session (no quit) DID auto-send. CODER's
> own smoke-readiness probe flagged this exact case: '5/6 cold spawns
> clean; the one anomaly was a pathological Claude boot stall (10-second
> silence) — Claude-internal state we can't fix from the sidecar.' Jose's
> machine consistently lands in that >3s-boot regime."

N2.1.5 fix used a chunk-gap heuristic: after writing the paste, watch for
pty silence; commit on `submitDelayMs` quiet OR at `submitMaxWaitMs` hard
deadline (3 s). Jose's machine routinely hits boot-stalls beyond 3 s, so
the hard deadline fires before Claude's TUI is paste-ready.

"Pathological" was my framing; Jose's report makes it clear this is his
common case. The fix needs a signal, not a bigger timer.

## §2 — Candidate signals considered

Dispatch §2 Task 1 listed three candidates (PM-patched 2026-04-22):

### §2.1 — PRIMARY (dispatch): Claude's `\x1b[?2004h` bracketed-paste-mode enable

Theory: Claude Code emits `\x1b[?2004h` when its Ink TUI starts accepting
paste input. Byte-exact upstream-stable pattern.

**Forensic result: REFUTED.** Scanned all N2.1.5 cold-probe dumps
(`/tmp/n2.1.5-cold-[1-3]-pm.raw`):

```
=== /tmp/n2.1.5-cold-2-pm.raw (26047 bytes) ===
  BP_ON (\x1b[?2004h): 1x at [19]
=== /tmp/n2.1.5-cold-3-pm.raw (26265 bytes) ===
  BP_ON (\x1b[?2004h): 1x at [19]
```

Byte 19 is zsh's prompt-init emission, not Claude's. Claude does NOT
re-emit bracketed-paste-on: Ink detects the terminal already has it
enabled (from zsh) and skips re-emission. PM's §0.1 reality-check
predicted this class of failure; it confirms here.

### §2.2 — SECONDARY (dispatch): boot-complete text pattern

Dispatch suggests `"Welcome to Claude"`, `"/effort"` footer, model banner.

**Forensic scan** (cold-2-pm.raw, 26047 bytes):

| Pattern | First byte | First chunk time |
|---|---|---|
| `"Opus"` | 1276 | t+770 ms |
| `"Opus 4.7"` | 2896 | t+2105 ms |
| `"/effort"` | 2845 | t+2105 ms |
| `"paste again"` | 3189 | t+2105 ms |
| `"shift+tab"` | 2793 | t+2105 ms |
| `"auto mode"` | 25940 | past-probe (too late) |
| `"to cycle"` | 25975 | past-probe (too late) |

`"/effort"` and `"paste again to expand"` at t+2105ms represent Claude's
input-field render — the TUI reaching "ready for paste" state.

**Concern:** content-pattern-matching violates OS §24.1 (upstream glyph
drift risk). If Anthropic renames `/effort` or "paste again" text, fix
breaks silently. But `/effort` is a slash-command name — very stable;
"paste again" is terminal input-state chrome that rarely changes.

### §2.3 — CHOSEN — ≥ 2 OSC title updates (structural, content-agnostic)

Claude Code uses Ink's built-in Spinner component which updates the
terminal title via OSC 0 / OSC 1 / OSC 2 escape sequences repeatedly
during boot and TUI active states. Each update is a distinct escape-
sequence emission the sidecar can count.

**Forensic scan** (cold-2-pm.raw):

```
=== OSC title emissions (\x1b]0; → BEL) ===
Count: 6
Position 1: byte 60, chunk t+742 ms
   content: \x1b]0;\xe2\x9c\xb3 Claude Code\x07
   (✳ U+2733 — initial launch banner)
Position 2: byte 3242, chunk t+2105 ms
   content: \x1b]0;\xe2\xa0\x82 Claude Code\x07
   (⠂ U+2802 — Ink Spinner first Braille frame)
Position 3-6: subsequent Braille frames as Spinner animates
```

**Key observation:** the SECOND OSC title emission fires at t+2105 ms —
exactly when `/effort`, `Opus 4.7`, and `paste again` text ALSO appear in
the same chunk. All these are coincident with "TUI ready for paste."

The signal "≥ 2 OSC title emissions since pty spawn" is:
1. **Structural** — counts escape-sequence frames, not text content. Upstream
   text renames don't affect it.
2. **Upstream-stable** — OSC title updates are terminal-integration
   primitive. Ink, chalk, ora, and every TUI framework uses them. If
   Claude Code switches spinner libraries, the title-update mechanism
   remains.
3. **Reliable** — appears in every successful N2.1.5 cold-probe dump.
   Zsh does not emit OSC 0 in our config (would conflict here — verified
   by looking at byte 60+ content starting with "Claude Code" string
   which is Claude-emitted).
4. **Easy to detect** — regex `\x1b\][012];[^\x07\x1b]*(\x07|\x1b\\)`
   matches any OSC title update with either BEL or ST terminator.

### §2.4 — Alternative considered: Braille byte detection

`\xe2\xa0\x80-\xbf` (Braille U+2800-U+283F) is used by Ink's Spinner.
First occurrence at byte 3245 in cold-2 (same as 2nd OSC title).

**Rejected vs. §2.3:** equivalent signal timing but slightly more
brittle — if Claude's spinner switches to dots, asterisks, or ANSI
cursor animation, Braille-byte detection breaks. OSC title counting
is spinner-library-agnostic.

## §3 — Fix shape (for Task 2)

Replace N2.1.5's chunk-gap + hard-deadline state machine with:

1. **`waitForClaudeReady(handle, onReady)` helper.** Parses pty output
   byte stream; counts OSC title emissions. Invokes `onReady` callback
   when count reaches `READY_OSC_COUNT` (default 2).
2. **Bootstrap injection sequence becomes:** pty spawn → `claude\n` on
   zsh's OSC 133 A → wait for Claude-ready signal → bracketed-paste write
   + `\r`.
3. **30-s hard timeout fallback.** If signal never fires, emit
   `system:warning` event ("Claude boot signal not observed within 30 s;
   attempting bootstrap write anyway — report to Jose if bootstrap fails")
   and proceed with the write. Better to attempt-and-fail visibly than
   block the session.

**State machine changes:**

```
Before (N2.1.5):
  wait-for-zsh-prompt → A marker → claude\n → wait-for-claude-ready
  wait-for-claude-ready → 800ms quiet OR hard deadline → write paste →
  wait-for-paste-quiet → chunk-gap OR hard deadline → \r → done

After (N2.1.6):
  wait-for-zsh-prompt → A marker → claude\n → wait-for-claude-ready
  wait-for-claude-ready → OSC title count ≥ 2 OR 30s timeout → write paste →
  wait-for-paste-quiet → chunk-gap OR 3s deadline → \r → done
```

The `wait-for-paste-quiet` stage is unchanged — it already works on
successful cases. Only `wait-for-claude-ready` swaps its signal.

**Post-paste quiesce retained** — after bracketed-paste write, waiting for
Claude to render the paste placeholder before `\r` is still the right
pattern.

## §5 — What this evidence does NOT cover

- **Whether Claude Code emits a DIFFERENT number of OSC title updates
  than 2 on some variants.** If Ink's spinner rate changes, the COUNT
  still exceeds 2 quickly (frames cycle every ~80 ms). No risk.
- **Whether future Claude Code versions remove the OSC title update
  feature entirely.** Unlikely — OSC title is a universal terminal
  integration feature even non-TUI apps use (zsh prompts, tmux, VS Code
  tasks). If removed, we fall back to the 30-s timeout + system:warning.
