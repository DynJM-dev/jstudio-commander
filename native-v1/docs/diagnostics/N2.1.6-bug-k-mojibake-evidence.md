## §1 — Symptom (Jose N2.1.5 smoke §3 Step 11)

> "Text fills with `â` / `â¢` / `âµ` / `â³` mojibake. Progressive
> degradation: 'better at first, but then it got worse and worse. When I go
> from our session to another it gets worse. Same with me restarting the
> app.'"

Signature pattern: a single character becomes `â<something>`. Textbook
UTF-8-decoded-as-Latin-1 mojibake: a UTF-8 byte 0xE2 (start of 3-byte
sequences like em-dash U+2014 = `0xE2 0x80 0x94`, or bullet U+2022 =
`0xE2 0x80 0xA2`) renders as Latin-1 codepoint U+00E2 = `â`.

Progressive degradation on session-switch / app-restart tells us the
corruption **accumulates through a persistence boundary**. Live-streamed
content renders correctly; restored-from-DB content corrupts. That
pinpoints the **scrollback round-trip path** as the culprit — not the
live pty stream.

## §2 — The scrollback round-trip path

Writer side (`apps/frontend/src/components/TerminalPane.tsx:152`):

```ts
async function flushScrollback(sessionId, serialize, opts) {
  const raw = serialize.serialize();                    // xterm returns a JS string (UTF-16)
  if (!raw) return;
  const blob = btoa(unescape(encodeURIComponent(raw))); // UTF-8 → base64
  await fetch(`/api/sessions/${sessionId}/scrollback`, {
    method: 'PATCH',
    body: JSON.stringify({ blob }),
    ...
  });
}
```

- `serialize.serialize()` → JS string. Multi-byte chars (`"—"`) are
  single JS codepoints (`U+2014` one `.charCodeAt(0)`).
- `encodeURIComponent("—")` → `"%E2%80%94"` (percent-encoded UTF-8 bytes).
- `unescape("%E2%80%94")` → `"\xE2\x80\x94"` (latin-1 string; each char is one UTF-8 byte).
- `btoa("\xE2\x80\x94")` → `"4oCU"` (base64 of 3 raw bytes `0xE2 0x80 0x94`).

**Writer produces base64 of UTF-8 bytes.** Correct.

Sidecar backend (`apps/sidecar/src/routes/scrollback.ts:59`):

```ts
let decoded = Buffer.from(blob, 'base64');         // Buffer of raw bytes
if (decoded.length > MAX_SCROLLBACK_BYTES) {
  decoded = decoded.subarray(decoded.length - MAX_SCROLLBACK_BYTES);
  truncated = true;
}
await db.drizzle
  .update(sessions)
  .set({ scrollbackBlob: decoded, updatedAt: now })   // BLOB column
  ...
```

- `Buffer.from(base64, 'base64')` → raw UTF-8 bytes preserved.
- Stored as BLOB (better-sqlite3 preserves bytes verbatim).
- Reader (same file, line 44-47): `buf.toString('base64')` → base64 of same raw bytes.

**Backend preserves bytes unchanged.** Correct.

Reader side (`apps/frontend/src/components/TerminalPane.tsx:119-120`):

```ts
const res = await httpJson<{ blob: string | null }>(...);
if (res.blob) {
  const decoded = atob(res.blob);   // latin-1 string (each char = 1 UTF-8 byte)
  term.write(decoded);              // ⚠ BUG: writes as UTF-16 string
}
```

- `atob(b64)` → the browser-native base64 → binary-string path. Returns
  a JS string where `charCodeAt(i)` equals the i-th byte (0-255).
- `term.write(binaryString)` → **`term.write` treats JS strings as
  UTF-16 codepoints.** Each UTF-8 byte becomes its Latin-1 codepoint.
  UTF-8 byte `0xE2` becomes U+00E2 = `â` rendered verbatim.

**Reader is the mojibake source.** The binary string from `atob` is NEVER
UTF-8-decoded back to a JS string; it's shoved into xterm as if it
were Unicode codepoints.

Progressive degradation explained: each save-restore cycle runs the
round-trip. Post-restore buffer already contains mojibake (`â`). Next
`serialize.serialize()` captures that mojibake as its new source
string. Base64-encoded `â`'s UTF-8 bytes (`0xC3 0xA2`) get persisted.
Next restore mojibakes AGAIN on the already-mojibake bytes.

## §3 — Fix shape (for Task 2)

Replace the reader with the bytes-direct path. Per G13 sub-agent audit
(logged in §4 + xterm.js [Encoding guide](https://xtermjs.org/docs/guides/encoding/)):
**`term.write()` accepts `Uint8Array` and UTF-8-decodes it internally.**
No JS-string round-trip needed.

```ts
// N2.1.6 Bug K fix. Before:
//   const decoded = atob(res.blob);
//   term.write(decoded);                     // ⚠ binary string → UTF-16 mojibake
// After:
const bytes = Uint8Array.from(atob(res.blob), (c) => c.charCodeAt(0));
term.write(bytes);                            // xterm UTF-8-decodes bytes internally
```

**Why this is correct:**

1. `atob(blob)` → binary string containing UTF-8 bytes as `charCodeAt` values.
2. `Uint8Array.from(str, c => c.charCodeAt(0))` → canonical idiom per
   MDN to extract raw bytes from an `atob` binary string. Produces a
   `Uint8Array` of the original UTF-8 bytes.
3. `term.write(Uint8Array)` → xterm.js 5.x uses its internal UTF-8
   decoder on byte input (xtermjs/xterm.js PR #1904, documented in
   xterm.js encoding guide). Em-dash `0xE2 0x80 0x94` correctly decodes
   to U+2014 `—`.

**No change needed** on the writer side — `btoa(unescape(encodeURIComponent(raw)))`
correctly produces base64 of UTF-8 bytes. No change needed on the sidecar —
`Buffer` preserves bytes. The bug is only at the reader's string-vs-bytes
interface.

**Zero-dep change** — uses native `atob` + `Uint8Array.from`. G12 N/A.

## §6 — What this evidence does NOT cover

- **Whether mojibake can occur in the LIVE (non-scrollback) path.**
  Live path writes `term.write(ev.data)` where `ev.data` is a JSON-decoded
  string (already UTF-16). This path is correct. If Jose sees mojibake
  BEFORE any restore happens, there's a separate bug — but his report
  describes progressive degradation through save/restore cycles, not
  live corruption.
- **xterm.js serialize-addon behavior with emoji / surrogate pairs.**
  Scope stays narrow — UTF-8 round-trip correctness is the fix. Emoji
  rendering is a v5+ surface that works with the same bytes-direct
  path.
- **The `decoded.subarray(decoded.length - MAX_SCROLLBACK_BYTES)`
  truncation at byte boundary.** Sub-agent confirmed xterm.js parser is
  robust to leading-edge truncation; not a contributor to Bug K's
  progressive degradation. Defer.
