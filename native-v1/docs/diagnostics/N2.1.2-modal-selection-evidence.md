## §2 — Layer of the failure (per SMOKE_DISCIPLINE.md §2)

The smoke stack covered in these steps:

```
USER → webview render → React event handler → component state (useState) →
  React effect (useEffect) → re-render → input `value` prop reconciliation
```

The webview fetch layer (N2.1.1 fix) is NOT involved — Jose's Step 7
already exercised dropdown-opens-and-populates-from-/api/session-types,
so the webview → sidecar path works.

Layer identified: **React component-state + effect-dep-array layer**.
Specifically: a `useEffect` whose dep array includes an unstable
reference, causing the effect to fire every render and reset state
the user just set.

## §3 — Root cause (code-read + library docs)

`apps/frontend/src/components/NewSessionModal.tsx:36-43` holds:

```tsx
// Reset on open.
useEffect(() => {
  if (open) {
    createMutation.reset();
    setProjectPath('');
    setSessionTypeId('pm');
  }
}, [open, createMutation]);
```

### 3.1 — Evidence A: `createMutation` reference is unstable

`const createMutation = useCreateSession()` → `useMutation(mutationFn, opts)`.
TanStack Query v5's `useMutation` returns a result object constructed on
every hook call. The individual handles (`mutate`, `mutateAsync`,
`reset`) are stable (React-stable callbacks from `useCallback` /
mutation-observer ref), but the top-level object wrapping them is NOT
wrapped in `useMemo`. A fresh reference is produced each render.

Reference: TanStack React Query v5 source at
`packages/react-query/src/useBaseMutation.ts` — the `useSyncExternalStore`
consumer returns a plain object each render; no outer memoization.

Consequence: `[open, createMutation]` dep array compares the previous
render's `createMutation` identity to the current render's identity via
`Object.is`. They differ. `useEffect` fires. Body runs. `setProjectPath('')`
and `setSessionTypeId('pm')` wipe whatever the user just set.

### 3.2 — Evidence B: the fire-every-render cycle

1. User clicks + New session → `open` flips false → true → effect fires
   (state triggers: resets are intentional on first-open).
2. Modal renders. `createMutation` = object ref A.
3. User clicks a project entry in the picker. `onChange(item.path)` fires
   → `setProjectPath(item.path)`. Render scheduled.
4. React re-renders. `createMutation` = object ref B (fresh from
   `useMutation`). `[open, createMutation]` dep array sees B ≠ A. Effect
   fires again. `setProjectPath('')` wipes the path the user just
   selected in step 3.
5. Render cycle continues. User sees: picker dropdown closes (the
   picker's internal setOpen(false) fired in commitPath before the
   wipe), but input value is empty. Symptom matches Step 8 exactly.

Same cycle for session-type: user picks Coder → `setSessionTypeId('coder')`
→ re-render → effect sees new createMutation ref → fires
→ `setSessionTypeId('pm')` wipes it back. Symptom matches Step 9
exactly.

### 3.3 — Evidence C: the second effect is NOT affected

`apps/frontend/src/components/NewSessionModal.tsx:46-49`:
```tsx
useEffect(() => {
  const match = typesQuery.data?.find((t) => t.id === sessionTypeId);
  if (match) setEffort(match.effortDefault);
}, [sessionTypeId, typesQuery.data]);
```

`typesQuery.data` is memoized by TanStack Query — structural sharing
keeps the same reference across renders when contents haven't changed.
So this effect only fires when `sessionTypeId` changes or when the
server's session-types data actually changes. No reset-on-render
pathology here.

### 3.4 — Evidence D: the third effect is also NOT affected

Lines 52-61 (`sessionTypeId === 'raw'` default-cwd pre-fill) deps are
`[sessionTypeId, rawCwdPref.data?.value]`. Scalar + eslint-disabled
missing `projectPath` (intentional — `if (!projectPath)` guard).
Both deps are stable. Safe.

### 3.5 — Ruled out

- **Picker onMouseDown preventDefault swallowing the click**: the
  picker's row button fires `onCommit(item)` inside `onMouseDown`, which
  synchronously calls `onChange(item.path)`. We confirmed via code-read
  that `onChange` *is* called — the wipe happens after, in the modal's
  rogue effect.
- **Native <select>'s onChange not firing**: native selects fire
  `change` event on option-click which React translates to `onChange`.
  Confirmed via code-read: line 153 `onChange={(e) => setSessionTypeId
  (e.target.value as SessionTypeId)}`. The setter IS called; it's
  overwritten on next render by the rogue effect.
- **Task 3 monotonic setOpen scope too broad**: PM's arrival hypothesis.
  Code-read refutes — `onClick={() => setOpen(true)}` is only on the
  trigger input, NOT on the dropdown-item buttons. Items have
  `onMouseDown` only and commit via `onCommit → commitPath → onChange
  + setOpen(false)`. The monotonic setOpen(true) is correctly scoped.

## §4 — Fix strategy (implemented in Task 2, not this file)

Minimum-scope fix: drop `createMutation` from the effect's dep array.
The effect body's `createMutation.reset()` call is still valid — it
reads the current render's `createMutation.reset` (a stable callback
from `useMutation`), it just shouldn't *trigger re-runs* when the wrapper
object identity changes.

```tsx
useEffect(() => {
  if (open) {
    createMutation.reset();
    setProjectPath('');
    setSessionTypeId('pm');
  }
  // eslint-disable-next-line react-hooks/exhaustive-deps — `createMutation`
  // is a TanStack-Query-v5 mutation result whose object identity is NOT
  // memoized across renders. Including it in deps fires this reset on
  // every render, wiping user selections. We intentionally read its
  // .reset() inside the body (.reset is stable) but do not depend on
  // the outer wrapper's reference.
}, [open]);
```

A more robust alternative is a ref — `createMutationRef.current =
createMutation` outside the effect + call `.reset()` through the ref
inside. The eslint-disable with inline rationale is more idiomatic
React and already a pattern used elsewhere in this file (line 60).

## §6 — Layer-identified summary (G11 compliance)

- **Diagnostic layer:** React component-state + useEffect dep-array.
- **Not diagnosed at:** picker mousedown handler (ruled out in §3.5),
  webview fetch (N2.1.1 layer, held), HTML <select> default action.
- **Evidence gathered at:** static code read of NewSessionModal.tsx
  lines 37-43 + TanStack Query v5 useMutation memoization docs.
- **Root cause one-line:** unstable TanStack Query mutation object
  reference in a useEffect's dep array fires the reset on every
  render, wiping user selections.
- **Scope of the bug:** every `useState` the reset effect touches is
  at risk. This fix addresses `projectPath` and `sessionTypeId`
  simultaneously (one fix, two symptoms) — same root cause that
  caused Steps 8 + 9 to fail together.
